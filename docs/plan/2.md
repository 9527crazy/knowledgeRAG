# 第 2 天实施计划：数据层与模型客户端封装

## 1. 文档目的

本文档用于细化 [代码编写计划](../coding-plan.md) 中“第 2 天：数据层与模型客户端封装”的落地方案，目标是在第 1 天骨架已经稳定的基础上，把后续索引链路和查询链路共同依赖的数据层、模型客户端与健康检查能力先搭起来。

本文档同时参考：

- [需求文档](../requirement.md)
- [开发环境文档](../development-environment.md)
- [第 1 天实施计划](./1.md)

## 2. 当日目标

第 2 天重点不是实现完整业务，而是把“外部依赖可用、存储可初始化、脚本可检查”这三件事跑通。

当日必须完成的目标：

- 封装 `Ollama` HTTP 客户端
- 封装 `Qdrant embedded` 初始化逻辑
- 封装 `SQLite` 台账初始化逻辑
- 创建 `document_index` 表结构
- 提供统一健康检查脚本

当日明确不做的内容：

- 不实现文档扫描与解析
- 不实现切块逻辑
- 不实现 Embedding 主链路
- 不实现向量写入与删除流程
- 不实现查询、Prompt、SSE API

原则是先把“依赖能连、存储能起、状态可见”固定下来，再进入第 3 天之后的业务链路开发。

---

## 3. 设计决策

为减少后续返工，第 2 天先固定以下实现约定。

### 3.1 模块边界约定

- `src/llm/ollama-client.ts` 负责所有对 `Ollama` 的基础 HTTP 调用
- `src/store/qdrant.ts` 负责 `Qdrant embedded` 的初始化与集合检查
- `src/store/ledger.ts` 负责 `SQLite` 连接与表初始化
- `src/scripts/health-check.ts` 作为统一依赖检查入口

这样后续业务模块不需要自己处理底层初始化细节。

### 3.2 Ollama 客户端约定

第 2 天只实现最小客户端能力，优先支持以下场景：

- 请求 `Ollama` 服务连通性
- 查询本地模型是否存在
- 为后续 chat / embedding 请求预留统一调用入口

建议先提供：

- `createOllamaClient(config)`
- `checkOllamaHealth()`
- `checkModelExists(modelName)`

第 2 天不要求把所有接口都做全，但结构上要能支撑后续扩展。

### 3.3 Qdrant 初始化约定

第 2 天先只处理“本地可连接”和“集合可创建或可复用”。

建议约定：

- 默认只维护一个 collection
- collection 名称在第 2 天先固定为单值，例如 `knowledge_chunks`
- 向量维度不在第 2 天写死到业务逻辑中，建议通过健康检查或初始化参数显式传入

第 2 天不做向量写入，只做：

- 本地数据目录初始化
- Qdrant 客户端创建
- collection 存在性检查
- collection 初始化

### 3.4 SQLite 台账约定

第 2 天需要先固定 `document_index` 表结构，保证第 4 天状态机可以直接接入。

建议字段至少包括：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `source_path` | TEXT | 文件路径，唯一键 |
| `doc_id` | TEXT | 文档标识 |
| `content_hash` | TEXT | 内容哈希 |
| `chunk_ids` | TEXT | chunk ID 列表，JSON 字符串 |
| `status` | TEXT | `indexing` / `done` / `failed` |
| `error_msg` | TEXT | 失败原因 |
| `indexed_at` | TEXT | 最近索引时间 |
| `updated_at` | TEXT | 最近更新时间 |

建议首版直接在初始化时执行 `CREATE TABLE IF NOT EXISTS`，避免引入迁移框架。

### 3.5 健康检查脚本约定

`health-check.ts` 负责把第 2 天的三个关键依赖状态一次性输出：

- Ollama 是否可连接
- 目标模型是否存在
- Qdrant 是否可初始化
- SQLite 是否可打开并完成表检查

输出建议采用结构化 JSON 摘要，便于后续人工查看与脚本消费。

## 6. 实施步骤

### 6.1 安装第 2 天依赖

在 `backend/` 中补充第 2 天需要的最小依赖。

建议引入范围：

- `@qdrant/js-client-rest`
- Bun 内建 `bun:sqlite`

说明：

- 第 2 天当前实现以本地 `Qdrant` HTTP 服务为准，默认连接 `http://127.0.0.1:6333`
- 本地数据持久化目录统一使用 `backend/data/qdrant`
- 依赖只围绕 Ollama、Qdrant、SQLite，不提前引入 Web 框架

### 6.2 实现 Ollama 基础客户端

建议能力拆分如下：

- 使用第 1 天的统一配置读取 `ollama_base_url`
- 封装基础 `fetch` 请求
- 提供服务健康检查接口
- 提供模型存在性检查接口

建议验证目标：

- 可调用 `GET /api/tags` 或等价接口获取模型列表
- 可确认 `llm_model` 与 `embedding_model` 是否已下载

实现注意点：

- 请求失败时统一抛出 `AppError` 子类
- 日志中输出模块名与关键上下文，不打印冗余响应体

### 6.3 实现 Qdrant 初始化模块

建议模块职责：

- 解析 `qdrant_path`
- 初始化本地存储目录
- 创建 Qdrant 客户端
- 提供 `ensureCollection()` 能力

建议第 2 天固定以下最小参数：

- collection 名称：`knowledge_chunks`
- distance：`Cosine`

实现注意点：

- 如果向量维度暂时无法自动推导，可在健康检查阶段通过一个显式常量或配置值完成初始化
- 本地开发前需先启动 `Qdrant` 服务
- 失败时应给出明确错误，说明是目录、客户端连接还是 collection 初始化失败

### 6.4 实现 SQLite 台账模块

建议模块职责：

- 打开 `ledger_path`
- 执行 `document_index` 表初始化
- 提供最小查询接口，例如：
  - `initLedger()`
  - `getLedgerStatus()`

首版表结构应至少覆盖后续状态机所需字段，不要求现在就把所有增删改查都实现完整。

实现注意点：

- `source_path` 建议加唯一约束
- `chunk_ids` 第 2 天先用 JSON 字符串保存
- 统一记录 `updated_at`

### 6.5 实现健康检查脚本

脚本职责建议控制在以下范围：

1. 加载配置
2. 检查 Ollama 服务可达
3. 检查生成模型与 embedding 模型是否存在
4. 初始化 Qdrant 本地目录与 collection
5. 初始化 SQLite 台账表
6. 打印结构化检查结果

推荐输出示例：

```json
{
  "ollama": {
    "reachable": true,
    "llm_model_ready": true,
    "embedding_model_ready": true
  },
  "qdrant": {
    "ok": true,
    "collection": "knowledge_chunks"
  },
  "ledger": {
    "ok": true,
    "table": "document_index"
  }
}
```

## 8. 验收清单

第 2 天完成后，需逐项确认以下结果：

- 在 `backend/` 目录执行依赖安装无报错
- `bun run typecheck` 无报错
- `bun run health-check` 可执行

- 程序可识别 `llm_model` 与 `embedding_model` 是否存在
- 本地 `qdrant_path` 能生成或复用数据目录
- 本地 `ledger_path` 能创建 SQLite 文件并初始化 `document_index` 表

## 9. 风险与规避建议

第 2 天常见风险如下：

| 风险 | 表现 | 建议处理 |
| --- | --- | --- |
| Qdrant 接入方式选错 | SDK 可装但 embedded 不能用 | 以官方推荐接入方式为准，不自行猜测 |
| SQLite 方案与 Bun 不兼容 | 运行时报原生模块错误 | 优先选择 Bun 已验证可用的方案 |
| 健康检查职责过重 | 脚本里混入后续业务逻辑 | 只做依赖检查，不做索引或查询 |
| 错误输出不统一 | 调试时难判断失败点 | 所有失败都走统一错误模型与日志入口 |
| 模型未提前拉取 | Ollama 可连但模型不存在 | 健康检查明确区分“服务可达”和“模型就绪” |

---

## 10. 完成定义

如果第 2 天结束时满足以下三条，可以判定当天目标达成：

1. `Ollama`、`Qdrant embedded`、`SQLite` 的最小封装已经建立。
2. `document_index` 表结构已可初始化，后续状态机可直接接入。
3. `health-check` 脚本可一次性反馈依赖健康状态。

这意味着第 2 天真正交付的不是索引功能本身，而是后续第 3 到第 5 天主链路开发所需的运行底座。
