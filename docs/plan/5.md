# 第 5 天实施计划：Embedding 与入库主链路

## 1. 文档目的

本文档用于细化 [代码编写计划](../coding-plan.md) 中「第 5 天：Embedding 与入库主链路」的落地方案。在第 3～4 天已形成「扫描 → 解析 → 切块 → MD5 → 增量计划 → SQLite `document_index`」能力的前提下，本日将 **Ollama Embedding（bge-m3）**、**Qdrant 向量写入与删除（按 chunk UUID）** 与 **台账最终状态（`done` / `failed`）** 接通，完成**单文档端到端入库闭环**，为第 6 天检索模块提供可检索向量数据。

## 2. 当日目标

当日必须把「向量写入 Qdrant + 台账记录与增量语义一致」跑通；**不实现查询链路、HTTP API、文件监听**。

当日必须完成的目标：

- 封装对每个 `TextChunk` 的 Embedding 调用（模型名取自配置 `embedding_model`，默认对齐 `bge-m3`）
- 请求失败按需求 **重试最多 3 次**；仍失败则将该文档整条索引标记为 `failed`，并写入 `error_msg`
- **同一文档内 chunk 串行** Embedding（避免并发压垮 Ollama）
- 将向量的 **Point ID** 与现有设计对齐：使用 chunk 自带的 UUID（`chunk_id`），写入与删除均使用该 ID，与台账 `chunk_ids` JSON 数组一致
- 按 [需求文档](../requirement.md) §7.1 组装 **Payload**（字段与现有 `ParsedDocument` / `TextChunk` / 配置可对齐：`doc_id`、`source_path`、`source_name`、`doc_title`、`doc_type`、`section_title`、`chunk_text`、`chunk_index`、`char_count`、`indexed_at` 等；其中 `chunk_index`、`total_chunks` 若暂未实现可先预留或填当前可知值）
- 扩展 `**vector-repository`**（或等价模块）：对上提供 `upsertChunks`、`deleteByChunkIds`（批量）等封装，对下使用 `@qdrant/js-client-rest`（与现有 [qdrant.ts](../../backend/src/store/qdrant.ts) 创建的 Collection 一致）
- 编写 `**index-document**`（或同名）编排：串联 `parseScannedFile` → `chunkDocument` → Embedding → Qdrant upsert → `ledger.upsertDocument`，并与第 4 天 `**buildIndexPlan**` 的输出类型衔接（至少支持 `add` / `update`；`skip` 不写向量；`remove` **删除 Qdrant 点**并 `**deleteBySourcePath` 台账**）

当日明确不做的内容：

- 查询侧 Embedding、Top-K、相似度过滤（第 6 天）
- Prompt、`/api/chat`、SSE（第 7～8 天）
- 文件监听与异步队列（可作为后续补强）

## 3. 设计决策

### 3.1 模块边界约定（与 coding-plan 一致）


| 模块        | 路径                                       | 职责                                                        |
| --------- | ---------------------------------------- | --------------------------------------------------------- |
| Embedding | `backend/src/ingest/embedder.ts`         | 输入文本，调用 Ollama Embedding API，返回浮点向量；内置重试                  |
| 向量仓储      | `backend/src/store/vector-repository.ts` | Chunk UUID 作为 point id upsert / 按 ID 批量删除；Payload 结构与需求对齐 |
| 入库编排      | `backend/src/ingest/index-document.ts`   | 单文档或计划驱动的完整入库；更新前删旧向量；失败写台账                               |


现有 `[createOllamaClient](../../backend/src/llm/ollama-client.ts)` 可扩展 `embeddings` 方法，或在 `embedder.ts` 内直接使用 `fetch` 调用 `POST /api/embeddings`（与 [Ollama API](https://github.com/ollama/ollama/blob/main/docs/api.md) 一致）。**命名与错误类型**：延续 `OllamaError`。

### 3.2 Embedding 约定

- 模型：`AppConfig.embedding_model`，向量维度：`embedding_dimensions`，须与 Qdrant Collection **一致**（现有 `ensureCollection` 已按配置维度创建）。
- **串行**：同一文档 `for (chunk of chunks)` await embed，文档之间可并行（本日可选，默认串行更简单）。
- **重试**：单次请求失败指数退避或固定间隔可选；连续 3 次失败后向上抛出，由编排层写 `failed`。

### 3.3 Qdrant 写入约定

- **Point ID**：采用 `chunk_id`（UUID 字符串）；Qdrant 支持 UUID string 作为 id（与 SDK 版本对齐时可用 `uuid` 或字符串 id 字段，以实际 SDK 写法为准）。
- **Payload**：至少包含需求 §7.1 表中「第一阶段使用」列为 ✅ 的字段；`indexed_at` 使用 Unix 毫秒整数。
- **更新文档**：先从台账或 `previousRecord` 读取旧 `chunk_ids`，调用 `deleteByChunkIds`（或等价）删除旧点，再写入新点，避免幽灵向量。

### 3.4 台账与 status 语义（第 5 天收口）

建议第 5 天完成后：

- `**indexing`**：入库进行中（可选：开始处理时写入）
- `**done**`：向量已成功写入 Qdrant，且 `chunk_ids`、`content_hash`、时间与第 4 天约定一致或增强
- `**failed**`：`error_msg` 记录原因（含 Embedding 最终失败简述）

若第 4 天已将无向量场景标为 `done`，本日应在大纲或注释中说明：**最终 `done` 表示已向量化并落库**，与注释一致。

### 3.5 删除路径

- `**remove`**：仅删 Qdrant 中与旧 `chunk_ids` 对应的点；删台账记录（已实现 `deleteBySourcePath`）。

## 6. 实施步骤概要

1. **embedder**：`embedText(text: string): Promise<number[]>`（或命名 `embedChunks`），内部调用 Ollama，`model: config.embedding_model`，解析返回的 `embedding` 数组。
2. **vector-repository**：依赖 `AppConfig` + `QdrantClient`，实现 `upsertChunks(points)`、`deleteChunkIds(ids: string[])`。
3. **index-document**：输入 `ScannedFile` + 已有 `content_hash`（或由内部 hash）；执行 parse → chunk → embed → upsert → `ledger.upsertDocument`；失败 catch 写 `failed`。
4. **runIndexPlanExecution**（可选）：遍历 `buildIndexPlan` 结果，对 `add`/`update` 调入库，对 `remove` 调删除，对 `skip` 跳过。

| 风险                   | 建议                                                     |
