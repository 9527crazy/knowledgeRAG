## 3. 设计决策

### 3.1 模块边界与复用

第 9 天的核心原则是：**watcher 只负责“发现变化并排队”，索引仍复用已存在的增量能力**。

复用点（已存在）：

- 扫描：`backend/src/ingest/file-scanner.ts`
- hash：`backend/src/ingest/hash.ts`
- 增量计划：`backend/src/ingest/index-state-machine.ts` 的 `buildIndexPlan()`
- 执行计划：`backend/src/ingest/index-document.ts` 的 `executeIndexPlan()`
- 单文档索引/删除：`indexDocument()` / `removeDocument()`
- 台账：`backend/src/store/ledger.ts`
- 向量库：`backend/src/store/vector-repository.ts`

新增点（第 9 天建议落点）：

- `backend/src/ingest/watcher.ts`：目录监听与事件归一化（create/change/remove）
- `backend/src/ingest/bootstrap-index.ts`：启动时“全量扫描 + 增量计划 + 执行”的一次性补扫入口
- `backend/src/scripts/watch-index.ts`（可选）：CLI 形式启动 watcher，便于单独验收与调试

### 3.2 事件归一化与去抖（避免抖动）

文件系统事件天然抖动（保存一次会触发多次 change）。建议策略：

- **过滤**：仅处理 `sources[].include` 白名单后缀（`.md`/`.txt`），其他忽略
- **去抖窗口**：对同一路径在 200~500ms 内的多次事件合并成一次
- **事件优先级**：
  - 同一路径在窗口内同时出现 create+change → 视为 change
  - remove 优先于 change（以最后一次事件为准）
- **目录事件**：忽略目录本身变更，只关注文件

### 3.3 索引执行模型（单 worker + 队列）

第一阶段建议采用 **单 worker 串行** 执行，保证：

- 不压垮 Ollama（Embedding/LLM）
- 日志可解释（同一时刻只有一个索引任务在跑）
- 避免台账并发写导致的竞态

队列策略：

- watcher 只做 `enqueue(changedPaths)`（Set 去重）
- worker 每轮：
  - 基于最新磁盘状态执行一次全量扫描（或对集合内路径做“最小扫描”）
  - 计算 hashes
  - 用 `buildIndexPlan()` 生成计划
  - 用 `executeIndexPlan()` 执行（remove 优先）

### 3.4 失败语义与可观测性

要求：

- 任一文件索引失败时：台账 `status=failed` 且 `error_msg` 截断（已在 `indexDocument()` 做了截断）
- watcher 侧异常（例如 watch 失败、权限不足）必须明确日志：包含 `source_name`、`source_path`
- 每次执行一轮增量时输出摘要日志：计划条数、add/update/remove 数量、耗时

## 4. 实施步骤概要

1. **bootstrap-index**：实现启动补扫
  - 全量扫描 sources
  - 计算 hashes
  - 生成 index plan
  - 执行 plan
2. **watcher**：实现目录监听
  - 对每个 source 启动 watcher（递归策略与 OS 能力匹配）
  - 事件归一化、过滤与去抖
  - 将变更路径批量送入队列
3. **worker loop**：串行执行增量任务
  - 处理一批变化后，若期间又有新变化，继续下一轮
4. **异常补强**：补齐日志与边界处理
  - 例如：文件刚被删除导致 read/hash 失败；应能转为 remove 或跳过并记录

## 7. 完成定义

1. watcher 跑通：新增/修改/删除文件能触发对应增量索引更新。
2. 重启补扫：服务重启后可自动发现离线期间的变更并补齐索引。
3. 失败可定位：台账可看到 failed 状态与 error_msg，日志可复现原因。
4. 与 [第 9 天校验文档](../test/9.md) 的验收步骤一致，且可稳定复现。
