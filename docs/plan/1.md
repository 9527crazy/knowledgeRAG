# 第 1 天实施计划：项目初始化与基础骨架

## 1. 文档目的

本文档用于细化 [代码编写计划](../coding-plan.md) 中“第 1 天：项目初始化与基础骨架”的落地方案，目标是把当日工作拆到可直接执行的粒度，避免进入实现阶段后反复返工。

本文档同时参考：

- [需求文档](../requirement.md)
- [开发环境文档](../development-environment.md)

---

## 2. 当日目标

第 1 天只做“骨架”，不追求业务功能完整，重点是把后续 9 天要依赖的基础设施搭稳。

当日必须完成的目标：

- 初始化 `backend/` 下的 `Bun + TypeScript` 项目
- 建立统一且可扩展的目录结构
- 定义基础脚本，保证项目可以启动和自检
- 建立统一配置加载模块
- 建立基础日志与错误处理约定

当日明确不做的内容：

- 不接入 `Qdrant embedded`
- 不接入 `SQLite`
- 不接入 `Ollama`
- 不实现实际索引、检索、问答逻辑
- 不提前引入 API 框架和复杂中间件

原则是先让“项目可启动、配置可读取、结构可扩展”，再在后续天数逐步填充业务能力。

---

## 3. 设计决策

为减少后续改动，第 1 天先固定以下实现约定。

### 3.1 项目根目录约定

- 仓库根目录只保留文档和顶层说明
- 后端实现统一放入 `backend/`
- 后续依赖安装、服务启动、脚本执行都以 `backend/` 为工作目录

### 3.2 配置管理约定

- `backend/config.ts` 作为对外统一配置出口
- `backend/src/config/` 存放配置类型、默认值、加载逻辑
- 运行配置建议放在 `backend/app.config.json`
- 环境变量只作为覆盖层，不直接散落在业务代码中读取

建议采用以下优先级：

`默认值 -> app.config.json -> 环境变量覆盖`

这样做的好处是：

- 首次运行有默认值，不会因为缺少环境变量直接失败
- 本地调试可通过 JSON 文件集中管理
- 线上或特殊环境也可以用环境变量覆盖关键参数

### 3.3 基础公共模块约定

- `backend/src/common/logger.ts` 提供统一日志入口
- `backend/src/common/errors.ts` 提供统一错误类型
- `backend/src/index.ts` 作为最小启动入口

### 3.4 目录预留约定

虽然第 1 天不开发业务逻辑，但建议把后续模块目录先建好，避免第 2 天开始频繁改结构。

建议首日即创建：

- `src/config/`
- `src/common/`
- `src/ingest/`
- `src/query/`
- `src/llm/`
- `src/store/`
- `src/server/`
- `src/scripts/`

## 4. 工时拆分建议

按 4 小时有效开发时间估算，可拆分为以下节奏：

| 时间 | 任务 | 产出 |
| --- | --- | --- |
| 0.5 小时 | 初始化 `backend/` 项目 | `package.json`、`tsconfig.json` |
| 0.5 小时 | 创建目录骨架 | `src/` 各模块目录 |
| 1 小时 | 实现配置模块 | `config.ts`、`src/config/*` |
| 1 小时 | 实现日志与错误约定 | `src/common/logger.ts`、`src/common/errors.ts` |
| 1 小时 | 启动入口与脚本自检 | `src/index.ts`、基础脚本、启动验证 |

如果实际进度偏慢，优先保证“配置加载 + 启动成功”两项，其余内容可以先保持最小实现。

## 6. 实施步骤

### 6.1 初始化 Bun + TypeScript 项目

建议流程：

```bash
mkdir -p backend
cd backend
bun init
```

初始化后需要立刻做两件事：

- 将 TypeScript 配置切到严格模式
- 清理模板中与本项目无关的示例代码

第 1 天只保留最小依赖即可，建议控制在以下范围：

- `typescript`
- Bun 类型支持
- 配置校验库，例如 `zod`（可选，但推荐）

不建议在第 1 天引入：

- Web 框架
- ORM
- 测试框架
- 监控或日志平台 SDK

---

### 6.2 建立基础脚本

建议在 `backend/package.json` 中至少定义以下脚本：

| 脚本名 | 作用 |
| --- | --- |
| `dev` | 启动最小入口，验证项目可运行 |
| `start` | 生产式启动入口，先与 `dev` 保持一致 |
| `typecheck` | 执行 TypeScript 类型检查 |
| `check:config` | 仅读取并打印校验后的配置 |

推荐约定：

- `dev` 和 `start` 暂时都指向 `src/index.ts`
- `check:config` 只验证配置，不做任何外部依赖连接

这样第 1 天即可用以下命令完成验收：

```bash
cd backend
bun install
bun run check:config
bun run dev
```

---

### 6.3 实现统一配置模块

第 1 天最重要的产出之一，是把第一阶段所有关键配置项先定义出来，即使部分配置在后续几天才真正用到，也应该先完成 schema 和默认值。

建议配置项至少包括：

| 配置项 | 示例值 | 说明 |
| --- | --- | --- |
| `sources` | 见下方示例 | 知识源目录列表 |
| `chunk_size` | `500` | 文本块长度上限 |
| `chunk_overlap` | `80` | 相邻块重叠长度 |
| `min_chunk_length` | `50` | 过短文本过滤阈值 |
| `top_k` | `6` | 检索候选数 |
| `similarity_threshold` | `0.7` | 相似度阈值 |
| `ollama_base_url` | `http://localhost:11434` | Ollama 地址 |
| `llm_model` | `qwen2.5:7b` | 生成模型 |
| `embedding_model` | `bge-m3` | 向量模型 |
| `server_port` | `3000` | 服务端口 |
| `qdrant_path` | `./data/qdrant` | 向量数据路径 |
| `ledger_path` | `./data/ledger.db` | 台账路径 |

建议的职责拆分如下：

- `src/config/types.ts`：定义 `AppConfig` 类型
- `src/config/schema.ts`：定义配置校验规则
- `src/config/defaults.ts`：定义默认配置
- `src/config/load-config.ts`：负责读取 JSON、合并默认值、应用环境变量覆盖、执行校验
- `config.ts`：对外导出 `loadConfig()` 或单例 `config`

推荐配置示例：

```json
{
  "sources": [
    {
      "name": "default",
      "path": "./knowledge",
      "include": [".md", ".txt"],
      "recursive": true
    }
  ],
  "chunk_size": 500,
  "chunk_overlap": 80,
  "min_chunk_length": 50,
  "top_k": 6,
  "similarity_threshold": 0.7,
  "ollama_base_url": "http://localhost:11434",
  "llm_model": "qwen2.5:7b",
  "embedding_model": "bge-m3",
  "server_port": 3000,
  "qdrant_path": "./data/qdrant",
  "ledger_path": "./data/ledger.db"
}
```

实现注意点：

- 所有相对路径统一以 `backend/` 为基准解析
- `sources` 至少校验非空、路径非空、扩展名列表非空
- `chunk_overlap` 不能大于等于 `chunk_size`
- `similarity_threshold` 应限制在 `0` 到 `1` 之间
- 配置校验失败时，启动流程应直接失败并输出明确错误

---

### 6.4 建立日志约定

第 1 天不需要引入复杂日志系统，但必须统一日志入口，避免后续模块直接 `console.log` 满天飞。

建议日志能力先做到以下程度：

- 支持 `info`、`warn`、`error`、`debug`
- 默认输出时间戳、级别、模块名
- `error` 支持打印 `code`、`message`、`cause`

推荐日志格式示例：

```text
[2026-04-23T10:00:00.000Z] INFO bootstrap config loaded
```

建议规则：

- 业务模块记录“发生了什么”，不要在日志中拼接大段无结构文本
- 敏感信息不要直接写日志
- 第 1 天先使用控制台输出，后续如有必要再切换 JSON 日志

---

### 6.5 建立错误处理约定

建议第 1 天即定义基础错误模型，例如：

- `AppError`
- `ConfigError`
- `ValidationError`

统一字段建议：

| 字段 | 说明 |
| --- | --- |
| `code` | 稳定错误码 |
| `message` | 面向开发者的错误描述 |
| `cause` | 原始异常对象 |
| `details` | 结构化上下文信息 |

最低要求：

- 顶层启动入口捕获未处理异常
- 已知错误按统一格式输出
- 未知错误打印原始堆栈并设置非零退出码

这样到第 2 天开始接外部依赖时，错误行为就已经统一，不用边开发边返修。

---

### 6.6 建立最小启动入口

`src/index.ts` 的职责只需要做到以下三件事：

1. 加载并校验配置
2. 初始化日志上下文
3. 输出“项目骨架启动成功”的日志

第 1 天不要在启动入口中接入任何真实业务逻辑。最小成功标准是：

- `bun run dev` 可以正常退出或保持运行
- 启动时能清晰打印配置加载结果
- 配置错误时能给出明确报错

如果需要更直观的自检，也可以增加一个 `src/scripts/check-config.ts`，专门用于读取并打印配置摘要。

## 8. 验收清单

第 1 天完成后，需逐项确认以下结果：

- 在 `backend/` 目录执行 `bun install` 无报错
- 执行 `bun run typecheck` 无报错
- 执行 `bun run check:config` 能输出校验后的配置
- 执行 `bun run dev` 无报错
- `backend/config.ts` 可作为全项目统一配置入口被导入
- 目录结构与需求文档、开发环境文档保持一致
- 没有把外部依赖接入逻辑提前塞进第 1 天骨架

建议额外做一次人工检查：

- 路径基准是否统一
- 脚本命名是否清晰
- 公共目录命名是否一致
- 配置字段命名是否与需求文档完全一致

---

## 9. 风险与规避建议

第 1 天常见风险如下：

| 风险 | 表现 | 建议处理 |
| --- | --- | --- |
| 初始化模板过重 | 自动生成无关文件较多 | 只保留最小可运行骨架 |
| 配置项先天不统一 | 后续模块字段命名反复修改 | 首日就按需求文档完整定义 schema |
| 目录命名摇摆 | `common` / `shared` 混用 | 第 1 天确定一种并坚持到底 |
| 过早接入依赖 | 当天时间被 SDK 调试吃掉 | 严格限制第 1 天只做骨架 |
| 启动入口职责过多 | 第 2 天后改动面扩大 | 保持 `index.ts` 只做 bootstrap |

---

## 10. 完成定义

如果第 1 天结束时满足以下三条，可以判定当天目标达成：

1. `backend/` 已经是一个可执行的 `Bun + TypeScript` 项目。
2. 配置、日志、错误处理三类基础设施已经具备最小可用能力。
3. 后续第 2 天可以在不调整目录结构的前提下直接开始接入 `Ollama`、`Qdrant embedded` 和 `SQLite`。

这意味着第 1 天真正交付的不是“业务功能”，而是后续开发速度和稳定性的地基。
