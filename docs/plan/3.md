# 第 3 天实施计划：文档读取与解析模块

## 1. 文档目的

本文档用于细化 [代码编写计划](../coding-plan.md) 中“第 3 天：文档读取与解析模块”的落地方案，目标是在第 1 天项目骨架和第 2 天依赖检查已经具备的前提下，把知识源目录扫描、文件过滤和基础解析能力搭起来，为第 4 天切块与状态机提供稳定输入。

本文档同时参考：

- [需求文档](../requirement.md)
- [开发环境文档](../development-environment.md)
- [第 2 天实施计划](./2.md)

---

## 2. 当日目标

第 3 天重点不是做索引入库，而是把“能读什么文件、读出来是什么结构、如何给后续模块稳定喂数据”先固定下来。

当日必须完成的目标：

- 实现知识源目录扫描
- 支持 `md` 与 `txt` 文件过滤
- 实现 Markdown 标题解析
- 实现 TXT 基础解析
- 生成统一文档结构对象
- 提取 `doc_id`、`doc_title`、`source_path`、`section_title`

当日明确不做的内容：

- 不实现切块逻辑
- 不实现文件 hash 与增量状态判断
- 不实现 embedding 调用
- 不实现向量写入与台账更新
- 不实现查询与问答逻辑

原则是先把“输入文档被稳定归一化”为第一优先级，再进入第 4 天切块和第 5 天入库主链路。

---

## 3. 设计决策

为减少后续返工，第 3 天先固定以下实现约定。

### 3.1 模块边界约定

- `src/ingest/file-scanner.ts` 负责目录扫描与文件过滤
- `src/ingest/parsers/markdown.ts` 负责 Markdown 标题结构解析
- `src/ingest/parsers/text.ts` 负责 TXT 基础解析
- `src/ingest/types.ts` 负责统一输入与解析结果类型

这样第 4 天开始时，切块模块可以直接消费统一文档结构，而不关心具体文件类型。

### 3.2 扫描范围约定

目录扫描应完全以配置 `sources` 为准：

- 支持一个或多个来源目录
- 支持递归扫描子目录
- 仅处理 `include` 中声明的扩展名
- 对不支持的扩展名直接忽略

建议第 3 天先只支持：

- `.md`
- `.txt`

如果后续增加 PDF 或代码文件，应在第 1 阶段之外单独扩展。

### 3.3 统一文档结构约定

第 3 天先固定一个统一解析结果结构，建议至少包含：


| 字段              | 说明                  |
| --------------- | ------------------- |
| `doc_id`        | 文档唯一标识              |
| `doc_title`     | 文档标题                |
| `source_name`   | 来源别名                |
| `source_path`   | 文档绝对路径              |
| `relative_path` | 相对 source 根目录路径     |
| `file_type`     | `markdown` 或 `text` |
| `sections`      | 文档分段结果              |


其中 `sections` 中的每个 section 建议至少包含：


| 字段              | 说明             |
| --------------- | -------------- |
| `section_title` | 标题路径或目录路径      |
| `content`       | 该 section 原始文本 |
| `order`         | 文档内顺序          |


第 3 天不要求 section 直接等价于最终 chunk，但它必须足够稳定，供第 4 天切块输入使用。

### 3.4 Markdown 解析约定

根据需求文档，第 3 天 Markdown 解析至少做到：

- 识别 `#`、`##`、`###`
- 将最近一级标题作为 section 的语义标题
- 把子目录路径与文档标题层级统一拼成 `section_title`

建议约定：

- `doc_title` 默认取文件名去扩展名；如果文档首个一级标题明显存在，可优先使用一级标题作为展示标题，但不改变 `source_path`
- section 的 `section_title` 使用 `/` 连接目录层级与标题层级
- 空内容标题段不单独输出 section

### 3.5 TXT 解析约定

TXT 不具备标题层级，因此建议：

- `doc_title` 取文件名去扩展名
- `section_title` 取相对目录路径，不含文件名
- 根目录下文件的 `section_title` 为空字符串
- 整个文件作为单一 section 输出

### 3.6 `doc_id` 约定

第 3 天先要求 `doc_id` 稳定且可复现，不要求引入复杂全局注册。

建议优先按以下方式生成：

- 使用 `source_name + relative_path` 的稳定组合
- 再做安全编码或 hash，避免直接暴露路径中的特殊字符

这样后续第 4 天和第 5 天可以稳定复用同一文档标识。

| 时间   | 任务                 | 产出                           |

如果时间不足，优先保证：

2. 文件过滤正确
3. Markdown/TXT 至少能产生稳定解析结果

## 6. 实施步骤

### 6.1 定义 ingest 类型

建议在 `src/ingest/types.ts` 中先固定以下类型：

- `SupportedFileType`
- `ScannedFile`
- `ParsedSection`
- `ParsedDocument`

这样扫描模块与解析模块可以通过类型契约对接，而不是靠松散对象传递。

### 6.2 实现目录扫描模块

建议 `file-scanner.ts` 提供以下最小能力：

- 遍历一个或多个 source
- 基于 `recursive` 控制是否深入子目录
- 基于 `include` 过滤扩展名
- 输出标准化 `ScannedFile[]`

建议扫描结果至少包含：

- `source_name`
- `source_root`
- `source_path`
- `relative_path`
- `extension`
- `file_type`

实现注意点：

- 所有路径输出统一为绝对路径
- `relative_path` 统一以 source 根目录为基准
- 扫描顺序建议固定，例如按路径排序，便于后续测试与比对

### 6.3 实现 Markdown 解析模块

建议最小职责：

- 读取 Markdown 文件内容
- 识别标题层级
- 生成 section 列表
- 构建完整 `section_title`

解析时建议注意：

- 无标题内容也应保留，必要时归入默认 section
- 标题本身不应丢失，应与对应正文形成 section 语义
- section 顺序必须与原文顺序一致

### 6.4 实现 TXT 解析模块

建议最小职责：

- 读取文件内容
- 生成单 section 结构
- 提取目录路径作为 `section_title`

实现注意点：

- 不对 TXT 做标题猜测
- 不做额外切块
- 保持原始文本内容不被改写

### 6.5 串联统一入口

第 3 天建议提供一个统一入口函数，例如：

- `scanSourceFiles(config)`
- `parseScannedFile(scannedFile)`

统一入口的目标是让第 4 天状态机只需要关心：

1. 先拿到待处理文件列表
2. 再把单文件转成 `ParsedDocument`

而不需要了解 Markdown/TXT 的具体差异。

### 6.6 准备样例文件验证

建议至少准备 3 个本地样例文件用于人工校验：

1. 带多级标题的 Markdown
2. 位于子目录中的 Markdown
3. 普通 TXT 文件

第 3 天不要求把样例文件提交到仓库根文档区，但至少要在本地验证中覆盖这些场景。

## 8. 验收清单

第 3 天完成后，需逐项确认以下结果：

- 能扫描一个或多个配置目录
- 仅处理配置白名单中的文件类型
- 扫描结果中的路径信息正确且稳定
- Markdown 文档能提取标题层级信息
- TXT 文档能输出基础元数据与单 section 结果
- 统一文档结构中已包含 `doc_id`、`doc_title`、`source_path`、`section_title`
- 至少 3 个样例文件的解析结果符合预期

## 9. 风险与规避建议

第 3 天常见风险如下：


| 风险                | 表现                   | 建议处理                                   |
| ----------------- | -------------------- | -------------------------------------- |
| 扫描与解析耦合过重         | 第 4 天状态机接入困难         | 提前分离 `scan` 与 `parse` 两层               |
| `doc_id` 不稳定      | 后续台账难复用              | 用 `source_name + relative_path` 生成稳定标识 |
| Markdown 标题层级处理混乱 | `section_title` 不可预测 | 先只支持 `#`、`##`、`###`，保持规则简单一致           |
| TXT 过度“智能解析”      | 输出结构不稳定              | 第 3 天保持单 section 策略                    |
| 路径基准不统一           | 后续日志和台账混乱            | 一律输出绝对路径 + source 相对路径                 |


---

## 10. 完成定义

如果第 3 天结束时满足以下三条，可以判定当天目标达成：

1. 系统已经能从配置指定目录中稳定扫描到待处理文档。
2. Markdown 与 TXT 都可以被转换成统一的 `ParsedDocument` 结构。
3. 第 4 天切块模块可以在不关心文件类型细节的前提下直接消费解析结果。

这意味着第 3 天真正交付的不是索引能力，而是把“原始文件”转换为“可进入索引链路的标准化文档对象”。
