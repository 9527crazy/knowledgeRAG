## 3. 设计决策

### 3.1 模块边界


| 模块           | 路径                                    | 职责                                                                                                   |
| ------------ | ------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| Prompt 拼装    | `backend/src/query/prompt-builder.ts` | 把 `question + Source[]` 拼装成 `system + user` 两条消息；输出对外稳定的 `ChatMessage[]`                             |
| LLM Chat 客户端 | `backend/src/llm/chat-client.ts`      | 调用 Ollama `POST /api/chat`，解析 NDJSON 流并以 `AsyncIterable<string>` 形式吐出 token；非流式调用作为兜底                |
| Chat 服务      | `backend/src/service/chat-service.ts` | 编排：`runRetrieval` → 空结果短路 / Prompt 拼装 → `streamChat` → 统一事件流（`delta` / `sources` / `done` / `error`） |
| 演示脚本         | `backend/src/scripts/chat-demo.ts`    | 命令行问答，按事件类型分别打印；便于第 8 天 SSE 接线前的端到端验证                                                                |


边界要点：

- `prompt-builder` 不感知 LLM 与 HTTP，只做字符串拼接，便于单测。
- `chat-client` 不感知检索与来源，只把 `messages` 与 LLM 参数翻译成 Ollama 协议；`AsyncIterable<string>` 让上层在不依赖 SSE 库的情况下也能逐 token 处理。
- `chat-service` 是唯一编排层，负责「无召回短路」「资料不足兜底」「事件类型归一」三件事；第 8 天会把它包成 SSE，第 9 天的健康检查/状态接口也复用此层。

### 3.2 Prompt 模板（对齐需求 §6.2）

System prompt 关键约束：

- 仅依据「参考资料」回答，不得编造、不得推测
- 资料不足时直接回复"资料中未找到相关内容"或同义表述，并简要说明缺哪类资料
- 引用资料时使用 `section_title` 与 `source_path`，便于人审追溯

User prompt 结构：

```text
# 用户问题
{question}

# 参考资料
[1] section_title: {section_title}
    source_path: {source_path}
    score: {score}
    --
    {chunk_text}

[2] ...
```

实现要点：

- 每条 `Source` 用编号 `[1] [2] ...` 隔开，方便 LLM 引用与人工核对
- `chunk_text` 不再二次截断（已在第 6 天 demo 内做了 120 字截断仅用于打印；Prompt 用完整文本）
- 当 `Source[]` 为空时不应进入此分支：在 `chat-service` 中先短路；`prompt-builder` 直接抛 `ValidationError("空来源不应进入 Prompt 拼装")` 作为防御性边界
- 不携带 `chunk_id` / `doc_id` 等内部字段，避免污染上下文长度

### 3.3 Ollama Chat 流式协议

参考 Ollama `/api/chat` 文档：请求体支持 `model / messages / stream / options`；响应体在 `stream: true` 下为 NDJSON，每行 JSON 形如 `{ message: { role, content }, done }`，最后一行 `done: true` 通常带 `eval_count` 等统计字段。

实现要点：

- 使用 `fetch + response.body.getReader()` 读取字节流；用 `TextDecoder("utf-8")` 增量解码；按 `\n` 切行解析（最后一行可能不完整，需保留到下一个 chunk）
- 对每行尝试 `JSON.parse`：
  - 解析失败 → `OllamaError("Ollama Chat 响应行不是合法 JSON")`，附 `raw_preview`
  - 解析成功且 `message.content` 非空 → `yield content`
  - `done === true` → 停止迭代
- 超时统一用 `AbortSignal.timeout(180_000)`（生成比 embedding 长，给 3 分钟兜底）
- 失败重试：第一阶段不在 chat 链路重试（重试容易引发"半流式"重复输出），仅做明确错误抛出；上层 service 决定是否兜底
- LLM 参数：`temperature = config.llm_temperature`、`num_predict = config.llm_max_tokens`（Ollama 字段名）

### 3.4 事件流契约（为第 8 天 SSE 做准备）

`chat-service.streamAnswer(question)` 暴露 `AsyncIterable<ChatEvent>`，其中 `ChatEvent` 形状与需求 §8.1 SSE 事件一一对应：

```ts
type ChatEvent =
  | { type: "delta"; text: string }
  | { type: "sources"; items: Source[] }
  | { type: "done" }
  | { type: "error"; message: string };
```

事件顺序约定：

- 正常路径：`sources` → 多个 `delta` → `done`
  - `sources` 在生成开始前就先吐出，便于前端先渲染来源占位
- 无召回 / 资料不足：`sources`(items=[]) → 单个 `delta`(兜底文本) → `done`，**全程不调用 LLM**
- 异常路径：`error` → `done`

这样第 8 天的 SSE 路由层只做事件 → SSE 帧的格式映射，不再处理业务逻辑。

### 3.5 「资料不足」兜底策略

明确两种触发：

1. `RetrievalResult.empty === true`（第 6 天已实现）：直接吐出兜底文本，不调用 LLM
2. LLM 输出包含「无法回答 / 资料不足 / 不知道」等关键词：第一阶段不做后处理，由模型自行表达；将这一职责留给 Prompt（system 中明确指示）

兜底文本为常量字符串，统一中文：

```text
资料中未找到与该问题直接相关的内容，无法据此作答。可尝试调整问题表述，或检查知识库是否包含相关文档。
```

## 6. 实施步骤概要

1. **prompt-builder**：
  - 导出 `ChatMessage = { role: "system" | "user" | "assistant"; content: string }`
  - 导出 `buildChatMessages(question, sources): ChatMessage[]`
  - 导出 `EMPTY_FALLBACK_TEXT` 常量供 service 复用
2. **chat-client**：
  - 导出 `ChatClient` 接口与 `createChatClient(config)`
  - 方法：`streamChat(messages, options?): AsyncIterable<string>`；可选保留 `chat(messages, options?): Promise<string>` 作为非流式兜底
  - 解析 Ollama NDJSON 流；失败统一抛 `OllamaError`
3. **chat-service**：
  - 导出 `ChatEvent` 联合类型与 `createChatService(config, deps?)`
  - 方法：`streamAnswer(question): AsyncIterable<ChatEvent>`
  - 内部：`runRetrieval` → 空召回短路（emit sources([])、delta(兜底)、done）→ 否则 emit sources、`buildChatMessages` → `streamChat` → 逐 token 转 `delta` → done
  - 错误统一捕获并 emit `error` + `done`
4. **chat-demo 脚本**：
  - 命令行第一参作为 question；不传则用默认问题
  - 把 `delta` 直接 `process.stdout.write`，`sources` JSON 打印，`error/done` 打印一行
  - 退出码：成功 0；`error` 事件出现则 1

## 9. 风险与规避建议


| 风险                                   | 建议                                                                    |
| ------------------------------------ | --------------------------------------------------------------------- |
| Ollama NDJSON 行被切断                   | 增量解码 + `\n` 拆分，保留尾部不完整片段到下一个 chunk                                    |
| LLM 输出过长导致首 token 延迟超 3 s（§10 非功能需求） | 限制 `num_predict`、`top_k`，并在 service 层先 `emit sources` 让前端先渲染来源        |
| 模型自由发挥编造内容                           | system prompt 明确"仅根据参考资料回答 + 资料不足时如实回复"；不依赖事后过滤                       |
| 流式过程中异常（断网、模型卸载）                     | service 捕获 → emit `error` + `done`，避免悬挂 reader                        |
| 与第 6 天接口耦合过紧                         | service 通过 deps 注入 `runRetrieval` 与 `chatClient`，便于第 8 天接 HTTP 时复用与测试 |
