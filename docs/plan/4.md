# 第 4 天实施计划：切块与增量索引状态机

## 1. 文档目的

本文档用于细化 [代码编写计划](../coding-plan.md) 中「第 4 天：切块与增量索引状态机」的落地方案。在第 3 天已产出统一 `ParsedDocument`、第 2 天已具备 SQLite 台账表结构的前提下，本日将**文本切块**、**文件内容 hash**、**增量决策**与**台账读写**串联起来，为第 5 天 Embedding、Qdrant 写入与完整入库闭环提供确定的输入与状态。

本文档同时参考：

- [需求文档](../requirement.md)（§5.2 增量索引状态机、§5.4 文本切块）
- [开发环境文档](../development-environment.md)
- [第 3 天实施计划](./3.md)

## 2. 当日目标

当日必须把「从解析结果到可索引 chunk」「从磁盘文件到是否需重索引」这两条链路说清楚并实现；**不向 Qdrant 写入向量**（归属第 5 天）。

当日必须完成的目标：

- 实现递归字符切块逻辑，并与配置项对齐
- 支持 `chunk_size`、`chunk_overlap`、`min_chunk_length`
- 对每个切块生成稳定可持久化的唯一标识（建议 UUID），满足需求「切块前自行生成 UUID」
- 实现文件内容 hash（与台账比对用），算法与需求一致：**MD5**，作用于**原始文件字节**（与监听/扫描到的文件一致）
- 实现增量状态判断：**新增**、**跳过**（hash 未变）、**更新**（hash 变化）、**删除**（台账有记录但当前扫描集合中已无该文件）
- 扩展或封装 SQLite `document_index` 的读写，供状态机查询与更新

当日明确不做的内容：

- 不实现 Ollama Embedding 调用
- 不向 Qdrant 写入或删除向量（删除/更新流程中「按 `chunk_ids` 删向量」可在本日以**空实现或第 5 天再接线**的方式预留，但**状态判定与台账侧删除意图**必须在本日可验收）
- 不实现 HTTP API、文件监听（监听可作为后续可与全量扫描并列补充）
- 不实现查询与问答逻辑

## 3. 设计决策

### 3.1 模块边界约定

建议固定为以下文件职责（与 [代码编写计划](../coding-plan.md) 一致）：


| 模块   | 路径                                          | 职责                                                |
| ---- | ------------------------------------------- | ------------------------------------------------- |
| 切块   | `backend/src/ingest/chunker.ts`             | 输入 `ParsedDocument` + 索引参数，输出带元数据的 chunk 列表       |
| Hash | `backend/src/ingest/hash.ts`                | 文件内容 MD5（或封装 `crypto`），供增量判断                      |
| 状态机  | `backend/src/ingest/index-state-machine.ts` | 输入「当前扫描到的文件集合 + 各文件 hash」与「台账已有记录」，输出下一步动作类型及所需数据 |


可选：在 `backend/src/ingest/types.ts` 中增补 `IndexedChunk`、`IndexAction` 等类型，避免状态机与切块层之间松散对象传递。

### 3.2 切块输入约定

切块模块只消费第 3 天的 `**ParsedDocument`**：

- 按 **section 顺序**将 `sections[].content` 拼接成单一字符序列后再切分，或在每个 section 内分别切分后按全局顺序编号；两种方式择一并在本文档实施阶段**写死**，保证同一文档多次运行 chunk 序号与边界一致。
- 每个 chunk 至少携带：`chunk_id`（UUID）、`chunk_text`、`doc_id`、`source_path`、`relative_path`、`section_title`（可归并自当前 section）、`chunk_index`（文档内从 0 递增）。

### 3.3 递归字符切块规则（与需求对齐）

与 [需求文档](../requirement.md) §5.4 一致：

- 采用递归字符切割策略。
- 切割优先级：`\\n\\n` → `\\n` → `。` → `！` → `？` → （必要时再缩小粒度，具体实现可在代码内用同一有序分隔符列表固定）。
- 每个窗口长度受 `chunk_size` 上限约束；相邻块之间重叠 `chunk_overlap` 字符。
- 去掉长度 **严格低于** `min_chunk_length` 的块（边界条件：等于 `min_chunk_length` 是否保留须在代码中单点注释说明）。

### 3.4 文件 Hash 约定

- 对 `**source_path` 指向文件的原始字节**计算 MD5（十六进制字符串入库），与「用户改文档但未触发解析边界」的场景一致。
- `doc_id`（第 3 天定义的文档标识）仍用于向量 payload 与业务关联；**台账主键**当前实现为 `source_path`（见 `document_index`），状态机以路径 + `content_hash` 比对为主。

### 3.5 增量状态机约定

与需求 §5.2 对齐的逻辑抽象（实现时可合并分支，但验收要能覆盖 equivalent 行为）：


| 场景                            | 判定  | 产出动作（命名可自定）                            |
| ----------------------------- | --- | -------------------------------------- |
| 磁盘上存在文件，台账无记录                 | 新增  | 准备解析 → 切块 → … → 台账写入（第 5 天补向量）         |
| 磁盘上存在，台账存在且 `content_hash` 一致 | 跳过  | 不写台账或仅更新轻量字段（若有）                       |
| 磁盘上存在，台账存在且 hash 不同           | 更新  | 标记需替换 chunk；第 5 天删除旧向量并写入新向量           |
| 台账有路径，本次扫描集合中不存在该路径           | 删除  | 读取旧 `chunk_ids`，第 5 天删向量；本日至少删除台账记录或标记 |


**扫描集合**：建议直接复用第 3 天 `scanSourceFiles` 的结果，与台账中 `source_path` 全集做 diff。

### 3.6 台账字段与状态枚举

沿用 [ledger 初始化](../development-environment.md) 及现有 `[document_index](../../backend/src/store/ledger.ts)` 结构时，建议统一 `status` 取值（可与需求对齐微调）：

- `indexing`：进行中
- `done`：切块与台账关键字段已写入（若第 5 天才写向量，可在实施时拆成 `pending_embed` / `done`，但须在代码注释说明与第 5 天的衔接）

`chunk_ids` 建议使用 **JSON 数组字符串**存储 UUID 列表，与需求一致。

## 6. 实施步骤

### 6.1 增补类型

在 `types.ts`（或独立 `chunk-types.ts`，二选一避免单文件过大）中定义：

- `TextChunk` / `IndexedChunk`：至少含 `chunk_id`、`chunk_text`、`doc_id`、`chunk_index`、`section_title`、`source_path` 等。
- `IndexIntent` / `IndexAction`：枚举或联合类型描述 `skip` | `add` | `update` | `remove` 等。

### 6.2 实现 `hash.ts`

- 提供 `hashFileContent(absolutePath): Promise<string>`，返回 MD5 hex。
- 错误路径（文件不存在、非文件）抛出现有工程内统一错误类型或包装为 `LedgerError` / `AppError` 子类。

### 6.3 实现 `chunker.ts`

- `chunkDocument(parsed: ParsedDocument, options: ChunkOptions): TextChunk[]`，其中 `ChunkOptions` 来自 `AppConfig` 中的 `chunk_size`、`chunk_overlap`、`min_chunk_length`。
- 保证输出按 `chunk_index` 单调，且过滤后序号仍连续或可验收地留空（须在注释中说明）。

### 6.4 扩展台账访问层

在 `document_index` 上提供：

- 按 `source_path` 查询一行
- 插入/更新一行（含 `content_hash`、`chunk_ids`、`status`、`updated_at`）
- 删除一行
- 可选：`listAllSourcePaths()` 或与扫描集 diff 所需的最小集合

### 6.5 实现 `index-state-machine.ts`

- 输入：`AppConfig`、`ScannedFile[]`、各文件 `content_hash`（或由内部调 `hash.ts`）
- 内部：加载台账记录，与当前扫描结果集对比
- 输出：结构化结果列表，例如 `{ path, action, reason?, previousRecord? }[]`
- **本日不要求**调用 Ollama/Qdrant；若执行「模拟跑通」，可脚本化：只对 `add` 路径执行解析 + 切块并打印 chunk 数量。

## 8. 验收清单

与 [代码编写计划](../coding-plan.md) 第 4 天验收标准对齐：

- 长文本可稳定切成多个 chunk，且重叠与上限符合配置
- 长度低于 `min_chunk_length` 的块被过滤
- 同一文件内容未变化时，状态机判定为跳过，不进入重新切块与无意义台账更新
- 文件内容变化后，状态机判定为更新，且能拿到旧 `chunk_ids`（为第 5 天删向量做准备）
- 文件从磁盘消失且仍在台账中时，判定为删除流程，并能读出待删除的 `chunk_ids`
- `bun run typecheck` 通过

## 9. 风险与规避建议


| 风险                          | 建议                                          |
| --------------------------- | ------------------------------------------- |
| 切块边界不稳定导致 hash 未变但 chunk 变化 | 本日以文件 MD5 为唯一真理；切块仅由同一文件内容确定性导出             |
| 路径规范化不一致（相对/绝对）             | 台账统一存 `scanSourceFiles` 给出的绝对 `source_path` |
| 第 4 天与第 5 天 `status` 语义冲突   | 在代码注释与本文档中固定「done 是否表示已向量化」，必要时增加中间状态       |
| 台账 API 不足导致状态机臃肿            | 优先把 CRUD 沉到 `ledger.ts`，状态机只做决策             |

## 10. 完成定义

若同时满足以下条件，可判定第 4 天目标达成：

1. 从 `ParsedDocument` 到 chunk 列表的路径明确且可重复运行。
2. 文件级 MD5 与台账比对能正确驱动跳过 / 更新 / 新增 / 删除四类意图。
3. SQLite 台账可被状态机读取与更新，字段与需求文档字段语义一致（允许类型为 TEXT 存储 JSON）。
4. 第 5 天可直接在「更新/删除」分支接入 Qdrant 与 Embedding，无需推翻切块与 hash 设计。

验收操作步骤建议单独维护在 `docs/test/4.md`（与 [规则文档](../rule.md) 中「有 plan 则有 test」对齐）。
