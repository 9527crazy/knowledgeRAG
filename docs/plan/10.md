## 3. 设计决策

### 3.1 测试策略

- **优先纯函数与无外部依赖模块**：Markdown 解析、切块边界、`buildIndexPlan` 决策逻辑等，使用 Bun 内置 `bun:test`（或项目统一测试入口），避免 CI 强依赖本机 Ollama/Qdrant。
- **少量集成测试（可选）**：在本地环境具备 Ollama + Qdrant 时，可通过环境变量开关跳过；默认 `bun test` 在 CI 或离线场景仍可绿。
- **不追求 100% 覆盖率**：以「四类关键逻辑」为底线（见 [第 10 天校验文档](../test/10.md)）。

### 3.2 状态与重索引 API

- **数据来源**：台账 `document_index` 统计 `done` / `failed` / `indexing`；`total_chunks` 可通过 Qdrant collection info 或检索侧已有封装查询（以实现为准，需单次请求可接受延迟）。
- **与 watcher 的关系**：`POST /api/reindex` 触发后应走与 `runIndexCycle` 相同的索引主链路，避免第三套实现。
- **错误语义**：非法 body、未知 `doc_id` 返回 400 JSON；内部错误 500 JSON（与现有 `server/app.ts` 风格一致）。

### 3.3 文档收口

- README 增补：**最小演示路径**（`health-check` → `dev` → `curl /api/chat`）、**watcher 与 `WATCH_DISABLED`**、**常用脚本表**。
- 已知问题：用表格列出「现象 / 影响 / 规避 / 计划修复阶段」，避免口头散落。

## 5. 实施步骤概要

1. 选定测试目录约定，增加 `bun test`（或 `npm test` 别名）脚本。
2. 按优先级补四类单测；必要时为 Qdrant 访问层引入可注入 mock。
3. 实现 `GET /api/status`、`POST /api/reindex`，在 `[backend/src/server/app.ts](../../backend/src/server/app.ts)` 注册路由。
4. 更新 README 与（可选）`docs/known-issues.md`。
5. 按 [第 10 天校验文档](../test/10.md) 走一遍完整演示与清单打勾。
