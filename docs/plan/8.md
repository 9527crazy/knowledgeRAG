## 3. 设计决策

### 3.1 复用与模块边界

第 8 天的核心原则是：**HTTP 层只负责协议与连接管理，不重复业务编排**。

- 复用第 7 天的 `ChatService`（`backend/src/service/chat-service.ts`）作为唯一业务入口
- `server` 层只做：
  - 解析请求体并校验
  - 把 `ChatEvent` 映射为 SSE 帧
  - 设置正确的响应头、处理连接关闭

### 3.2 SSE 事件协议（对齐需求 §8.1）

事件类型与 payload 约定如下：

- `delta`：`{ text: string }`
- `sources`：`{ items: Source[] }`
- `done`：无数据
- `error`：`{ message: string }`

SSE 帧格式：

```text
event: delta
data: {"text":"..."}

```

说明：

- 每个事件后必须有一个空行
- `data` 必须是单行 JSON（不跨行）

### 3.3 路由与文件落点

建议产出与边界（与 `coding-plan` 对齐）：


| 模块      | 路径                                  | 职责                                                              |
| ------- | ----------------------------------- | --------------------------------------------------------------- |
| app     | `backend/src/server/app.ts`         | 创建并启动 HTTP 服务，注册路由                                              |
| SSE 编码  | `backend/src/server/sse.ts`         | 把 `{ event, data }` 编码为文本帧；提供便捷写入函数                             |
| chat 路由 | `backend/src/server/routes/chat.ts` | `POST /api/chat`：校验入参 → 调用 `createChatService(config)` → 输出 SSE |


路由建议：

- `POST /api/chat`：核心接口（SSE）
- `GET /healthz`：可选的轻量就绪检查（返回 200 + JSON），便于 curl 验证服务活着（不做依赖探测，依赖探测继续用 `bun run health-check`）

### 3.4 请求校验与错误语义

请求体：

```json
{ "question": "..." }
```

校验策略：

- `question` 缺失或不是 string → 400 + JSON 错误（非 SSE）
- `question.trim().length === 0` → 400 + JSON 错误（非 SSE）

服务端内部错误：

- 在 SSE 流中发出 `error` 事件 + `done`，并主动结束连接

## 6. 实施步骤概要

1. **sse.ts**：实现 `encodeSseEvent(event, data?)` 与 `writeSse(controller, event, data?)`，统一 JSON 序列化与换行规则。
2. **chat.ts**：实现 `POST /api/chat`：
  - 读取并校验 JSON body
  - 创建 `ChatService`，遍历 `streamAnswer(question)`：
    - `sources` → SSE `sources`
    - `delta` → SSE `delta`
    - `error` → SSE `error` + SSE `done` 并结束
    - `done` → SSE `done` 并结束
3. **app.ts**：注册路由与默认 404；提供 `startServer(config)`。
4. **联调**：用 curl 触发 `POST /api/chat`，确认事件顺序与字段；模拟无召回（调高 `SIMILARITY_THRESHOLD`）验证不会调用 LLM 也能返回 `delta + done`。

## 9. 风险与规避建议


| 风险               | 建议                                                                         |
| ---------------- | -------------------------------------------------------------------------- |
| SSE 连接被代理缓冲导致不流式 | 设置 `Cache-Control: no-cache`，并尽量 `flush`（Bun 下以持续 `controller.enqueue` 为主） |
| JSON 序列化跨行破坏 SSE | `data` 必须 `JSON.stringify` 为单行，禁止直接拼接多行文本                                  |
| 客户端断开导致写入报错      | 捕获 enqueue 错误并中止迭代，及时结束服务端生成                                               |
| 事件顺序不一致          | 以 `ChatService` 的事件顺序为准，HTTP 层只映射，不重排                                      |
