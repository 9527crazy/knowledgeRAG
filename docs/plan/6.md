## 3. 设计决策

### 3.1 模块边界约定（与 coding-plan 一致）

建议固定为以下文件职责：


| 模块           | 路径                                   | 职责                                                                     |
| ------------ | ------------------------------------ | ---------------------------------------------------------------------- |
| 查询 Embedding | `backend/src/query/embed-query.ts`   | 输入 question，输出 query vector（长度必须等于 `embedding_dimensions`）；失败按既有重试策略抛错 |
| 检索           | `backend/src/query/retriever.ts`     | 调用 Qdrant `search`，返回 Top-K ScoredPoint；并基于 `similarity_threshold` 做过滤 |
| 来源格式化        | `backend/src/query/source-mapper.ts` | 将检索结果映射为稳定的 `Source` 输出结构（供第 7～8 天复用）                                  |


说明：

- 查询侧 Embedding 可**复用第 5 天**的 `createEmbedder(config)`：它对任意文本执行 `POST /api/embeddings`，并对维度做校验。
- 检索侧不新增 store 层改动：直接在 `retriever.ts` 内创建 `QdrantClient` 调用 `search` 即可（与 `vector-repository.ts` 的 URL/collection 口径保持一致）。

### 3.2 Qdrant Top-K 与阈值过滤

约定：

- Top-K：使用 `config.top_k`
- 相似度阈值：使用 `config.similarity_threshold`
- 距离度量：Qdrant Collection 已按 `Cosine` 创建（见 `backend/src/store/qdrant.ts`）

实现策略：

- 调用 `client.search(collection, { vector, limit: config.top_k, with_payload: true, with_vector: false })`
- 阈值过滤两种方式择一（实现时二者可同时使用，保证行为更清晰）：
  - 在请求侧传 `score_threshold: config.similarity_threshold`（交给 Qdrant 过滤）
  - 在返回侧再做一次 `score >= config.similarity_threshold` 的过滤（作为兜底，避免 SDK/服务端行为差异导致漏过滤）

### 3.3 来源信息口径

来源输出应满足需求 §6.3 “来源引用”最低字段集：

- `source_path`
- `section_title`
- `score`

建议额外保留：

- `source_name`（来自 Payload）
- `doc_title` / `doc_type`（来自 Payload，便于调试与前端展示）

来源字段均从 Qdrant Payload 读取（见 `backend/src/store/vector-repository.ts` 的 `ChunkPayload` 定义，与需求 §7.1 对齐）。

### 3.4 “无结果时直接返回”的边界

当过滤后无结果时，必须：

- 返回一个明确的“未召回”结构（例如 `items: []` + `message`），或抛出可识别的业务错误码（由上层处理）
- 并且不触发第 7 天的 LLM 调用（即使上层已接入，也应能根据该返回结构短路）

## 6. 实施步骤概要

1. **embed-query**：
  - 导出 `embedQuery(config, question): Promise<number[]>`
  - 内部使用 `createEmbedder(config).embed(question)`
2. **retriever**：
  - 导出 `retrieveTopK(config, vector): Promise<ScoredPoint[]>`（或自定义轻量结构）
  - 调用 Qdrant `search`，返回带 `score` 与 `payload` 的候选
  - 执行阈值过滤
3. **source-mapper**：
  - 定义 `Source` 输出结构（至少含 `source_path`、`section_title`、`score`）
  - 从 payload 中读取并格式化
4. **无结果短路**：
  - 在 `retriever` 或一个薄的 `query` 编排函数中统一处理：过滤后空数组直接返回

## 8. 验收清单（与 coding-plan 第 6 天对齐）

- 输入问题可返回 Top-K 候选片段（至少包含 `chunk_text` 与来源信息）
- 相似度低于阈值的结果会被过滤（可通过调高阈值制造“无结果”场景验证）
- 无召回结果时，流程可直接返回且**不调用 LLM**
- 返回结果中包含 `source_path`、`section_title`、相似度（`score`）

详细步骤见 [第 6 天校验文档](../test/6.md)。

## 9. 风险与规避建议


| 风险               | 建议                                                        |
| ---------------- | --------------------------------------------------------- |
| 检索返回 payload 缺字段 | 第 6 天验收时必须确认 `with_payload: true` 且字段名与 `ChunkPayload` 一致 |
| `score` 语义与阈值不匹配 | Collection 使用 `Cosine`，阈值先按 0 \sim 1 经验值设置；以可复现实验校准为准     |
| 过滤后无结果但仍触发下游     | 统一在检索编排层做“空结果短路”，并在第 7 天接入时保持该短路逻辑不被绕过                    |
