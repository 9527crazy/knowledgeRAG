## 2. 校验范围

本次校验覆盖：

- Query Embedding：输入 `question` 生成向量，维度与 `embedding_dimensions` 一致
- Qdrant Top-K：按 `top_k` 返回候选
- 相似度阈值过滤：低于 `similarity_threshold` 的候选被剔除
- 来源格式化：至少包含 `source_path`、`section_title`、`score`
- 无结果短路：过滤后为空时能直接返回（且为第 7 天“不调用 LLM”提供可识别信号）

本次不校验：

- Prompt 拼装、LLM 调用与流式输出（归属第 7～8 天）
- HTTP API / SSE（归属第 8 天）
- 文件监听与自动增量更新（归属第 9 天）

## 6. Query Embedding 校验

### 6.1 校验目标

- 对非空 `question` 可返回长度为 `embedding_dimensions` 的数组
- 对错误模型名/无法连接 Ollama 的场景，可按既有重试策略失败并抛错（由上层处理）

### 6.2 建议操作

（若已提供演示脚本）运行一次 query embedding + 检索 demo，或在 REPL/脚本中直接调用 `embedQuery()`。

### 6.3 通过标准

- 维度与配置一致
- 异常情况能明确失败（不吞错、不返回空向量）

---

## 7. Qdrant Top-K 检索与阈值过滤校验

### 7.1 校验目标

- 检索请求能返回最多 `top_k` 条候选
- 候选含 `score`（相似度）与 payload（用于映射来源字段）
- `score < similarity_threshold` 的候选会被过滤

### 7.2 建议操作

1. 使用一个明显与知识库相关的 `question` 执行检索，确认能召回至少 1 条结果。
2. 调高 `SIMILARITY_THRESHOLD`（例如设为 `0.99`），再次执行同一问题，期望结果变少或变为 0。

### 7.3 通过标准

- Top-K 的上限行为符合 `top_k`
- 阈值过滤可被调参验证（有结果 / 无结果两种现象可解释）

---

## 8. 来源信息格式化校验

### 8.1 校验目标

检索结果映射后，每个来源条目至少包含：

- `source_path`
- `section_title`
- `score`

建议额外包含（若实现）：

- `source_name`

### 8.2 通过标准

- 字段齐备，且来源信息与 Qdrant payload 中字段一致（与需求 §6.3 对齐）

---

## 9. 无结果短路校验（为第 7 天做准备）

### 9.1 校验目标

当阈值过滤后无结果时：

- 能直接返回“无召回”的结果结构（例如 `items: []`）
- 上层能据此判断“不应调用 LLM”（第 7 天接线时必须保留该短路）

### 9.2 建议操作

- 通过调高 `SIMILARITY_THRESHOLD` 或使用无关问题制造无结果场景，观察输出结构。

### 9.3 通过标准

- 空结果场景可复现，且输出清晰、稳定

---

## 10. 完成定义

当以下全部满足时，可判定第 6 天校验通过：

1. `embed-query.ts`、`retriever.ts`、`source-mapper.ts` 已落地并能协同工作。
2. Top-K 与阈值过滤行为可通过调参复现并解释。
3. 返回结构中包含 `source_path`、`section_title`、`score`，满足第 7 天 Prompt 输入与来源引用的前提。
4. `bun run typecheck` 通过。
5. 与 [规则文档](../rule.md) 一致：存在 [第 6 天实施计划](../plan/6.md) 与本文件的成对关系。

若第 6 天代码已写完，但仍无法通过本文件中的校验项，则不能视为真正完成，也不宜进入第 7 天 Prompt 与回答生成开发。
