## 2. 校验范围

本次校验覆盖：

- 自动化测试：至少覆盖 **切块、解析、增量状态机、检索** 四类关键逻辑之一（四类全覆盖为通过标准）
- 对外接口：`GET /api/status`、`POST /api/reindex`（与需求字段语义一致）
- 文档：`README.md` 能指导新人完成本地演示
- 端到端演示：资料变更后问答可感知更新

本次不校验：

- 第二阶段能力（PDF、多轮、混合检索等）
- 生产级监控、鉴权、限流

## 4. 自动化测试

### 4.1 运行命令

以实现为准（计划默认为 Bun 测试）：

```bash
cd backend
bun test
```

### 4.2 通过标准

- 退出码为 `0`
- 至少存在并执行以下四类中的测试文件（名称可调整，但职责需对应）：
  - 切块（chunker）
  - 解析（markdown 或 parse 层）
  - 增量状态机（`buildIndexPlan` 或等价决策函数）
  - 检索（retriever 或 query 编排；允许 mock 外部依赖）

若仓库选择「集成测试默认跳过」，则须在本文档或 README 中写明开关；**未写明开关且默认失败**视为不通过。

## 5. API：`GET /api/status`

### 5.1 校验命令

服务启动后（`cd backend && bun run dev`，端口以配置为准，示例为 3000）：

```bash
curl -sS http://localhost:3000/api/status | head
```

### 5.2 通过标准

- HTTP 200
- 响应为 JSON，且包含需求 §8.2 所列字段（允许额外字段，但不得缺失）：
  - `total_docs`、`total_chunks`、`failed_docs`、`indexing_docs`、`last_indexed_at`、`sources`
- `sources` 为数组，元素至少含 `name`、`path`、`doc_count`

---

## 6. API：`POST /api/reindex`

### 6.1 校验命令（示例）

触发全部失败文档重索引（若当前无 `failed`，可先人为制造一条 `failed` 再调用，步骤以实现为准）：

```bash
curl -sS -X POST \
  -H "Content-Type: application/json" \
  -d '{}' \
  http://localhost:3000/api/reindex
```

指定 `doc_id`（可选）：

```bash
curl -sS -X POST \
  -H "Content-Type: application/json" \
  -d '{"doc_id":"<你的 doc_id>"}' \
  http://localhost:3000/api/reindex
```

### 6.2 通过标准

- HTTP 200
- 响应 JSON 含 `triggered`（数字）
- 调用后台账中对应文档状态可向 `done` 收敛或保持可解释的失败（需能在 `error_msg` 中定位）

## 7. README 与已知问题

### 7.1 README

打开 [README.md](../../README.md)，确认至少包含：

- 依赖：Bun、Ollama、模型名、Qdrant（若仍为独立进程需写明）
- 启动：`bun install`、`bun run dev`
- 常用脚本：`health-check`、`index:full`、`watch:index`、`query:demo`、`chat:demo`
- 环境变量：如 `WATCH_DISABLED`、`SERVER_PORT` 等（以实现为准）

### 7.2 已知问题

存在「已知问题」小节（README 内或独立 `docs/known-issues.md` 并在 README 链接），每条包含：现象、影响、规避方式、后续计划。

## 8. 端到端演示（必做）

目标：可向他人口述并复现一次完整故事。

建议路径：

1. 在知识目录新增或修改一篇 `md`，确认 watcher 或 `watch:index` 日志出现索引成功信息（与第 9 天一致即可）。
2. `curl` 调用 `POST /api/chat`，确认 SSE 中出现 `sources` 与 `delta`，且内容能引用到刚变更的文档要点。
3. 再次修改同一文档，重复步骤 2，确认回答能反映更新（或来源得分/片段变化可观察）。

---

## 9. 完成定义

当以下全部满足时，可判定第 10 天校验通过：

1. `bun test`（或项目约定测试命令）通过，且四类关键逻辑均有覆盖。
2. `GET /api/status`、`POST /api/reindex` 可用且字段语义与需求一致。
3. README 可独立完成演示；已知问题已记录。
4. 端到端演示路径可复现。
