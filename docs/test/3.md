# 第 3 天校验文档：文档读取与解析模块

## 1. 文档目的

本文档用于对齐 [第 3 天实施计划](../plan/3.md) 的验收方式，明确在“文档读取与解析模块”完成后，应该如何检查扫描、过滤与解析能力是否满足后续切块与索引链路的输入要求。

本文档主要回答：

- 第 3 天做完后要检查哪些内容
- 哪些现象代表解析能力已达标
- 哪些问题说明还不能进入第 4 天开发

本文档同时参考：

- [第 3 天实施计划](../plan/3.md)
- [代码编写计划](../coding-plan.md)
- [规则文档](../rule.md)

---

## 2. 校验范围

本次校验覆盖以下内容：

- 知识源目录扫描
- `md` 与 `txt` 文件过滤
- Markdown 标题解析
- TXT 基础解析
- 统一文档结构输出

本次明确不校验：

- 切块逻辑
- hash 计算
- SQLite 状态机读写
- embedding 调用
- Qdrant 写入

---

## 3. 前置条件

开始校验前，需满足以下条件：

- `backend/` 已存在并可执行 Bun 脚本
- 第 1 天与第 2 天基础能力已可用
- `backend/knowledge/` 或配置中的 source 目录内已准备样例文件

建议至少准备以下 3 类样例：

1. 带多级标题的 Markdown
2. 位于子目录中的 Markdown
3. 普通 TXT 文件

---

## 4. 文件产出校验

### 4.1 需要存在的文件

完成第 3 天实现后，至少应存在以下文件：

```text
backend/src/ingest/types.ts
backend/src/ingest/file-scanner.ts
backend/src/ingest/parsers/markdown.ts
backend/src/ingest/parsers/text.ts
```

### 4.2 校验方法

可执行：

```bash
find backend/src/ingest -maxdepth 3 | sort
```

### 4.3 通过标准

- 上述文件存在
- 模块职责与计划文档一致
- 未把切块、embedding 或入库逻辑提前混入解析层

---

## 5. 类型与基础检查

### 5.1 校验命令

```bash
cd backend
bun run typecheck
```

### 5.2 通过标准

- 命令退出码为 `0`
- ingest 模块新增类型和解析结果无类型错误

---

## 6. 文件扫描校验

### 6.1 校验目标

- 能扫描一个或多个 source
- 能按 `include` 白名单过滤
- 能正确处理递归目录

### 6.2 建议检查点

- 扫描结果中的 `source_path` 为绝对路径
- `relative_path` 与 source 根目录关系正确
- 不支持的扩展名不会进入结果
- 结果顺序稳定，重复执行不会乱序

### 6.3 通过标准

- `.md` 与 `.txt` 被正确识别
- 非白名单文件被正确忽略
- 子目录中的文件能被发现并保留正确相对路径

---

## 7. Markdown 解析校验

### 7.1 校验目标

- 能识别 `#`、`##`、`###`
- 能提取稳定的 `doc_title`
- 能生成带顺序的 section 列表
- 能构造符合规则的 `section_title`

### 7.2 建议样例

建议使用包含以下结构的 Markdown：

- 文档一级标题
- 至少一个二级标题
- 至少一个三级标题
- 标题前后都带正文

### 7.3 通过标准

- section 顺序与原文一致
- 标题层级不会丢失
- `section_title` 能正确拼上目录路径与标题路径
- 解析结果中的 `doc_id`、`doc_title`、`source_path` 存在且稳定

---

## 8. TXT 解析校验

### 8.1 校验目标

- TXT 文件能被解析为单一 section
- `doc_title` 取文件名去扩展名
- `section_title` 取相对目录路径

### 8.2 通过标准

- 根目录 TXT 的 `section_title` 为空字符串
- 子目录 TXT 的 `section_title` 为目录路径
- `content` 保留原始文本，不做额外切块或标题猜测

---

## 9. 统一结构校验

### 9.1 校验目标

无论输入是 Markdown 还是 TXT，输出都应能落到统一结构中。

### 9.2 最低字段要求

统一结构中应至少包含：

- `doc_id`
- `doc_title`
- `source_name`
- `source_path`
- `relative_path`
- `file_type`
- `sections`

每个 section 至少包含：

- `section_title`
- `content`
- `order`

### 9.3 通过标准

- 两类文件的结果结构一致
- 第 4 天切块模块无需再区分原始文件类型

---

## 10. 样例验证要求

### 10.1 建议最少样例数

至少验证以下 3 个样例：

1. 根目录 Markdown
2. 子目录 Markdown
3. 子目录或根目录 TXT

### 10.2 通过标准

- 3 个样例都能成功输出统一解析结果
- 至少人工确认一次 `section_title`、`doc_title`、`relative_path` 是否符合预期

---

## 11. 完成定义

当以下项目全部满足时，可判定第 3 天校验通过：

1. 扫描模块能稳定输出待处理文件列表。
2. Markdown 与 TXT 都能被转换为统一文档结构。
3. 统一结果已包含后续切块与索引所需的关键字段。
4. 至少 3 个样例文件已完成人工校验。
5. 文档结构继续满足“有 plan 就有对应 test”的规则。

如果第 3 天代码已写完，但仍无法通过本文件中的校验项，则不能视为真正完成。
