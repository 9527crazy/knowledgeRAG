# 第 1 天校验文档：项目初始化与基础骨架

## 1. 文档目的

本文档用于对齐 [第 1 天实施计划](../plan/1.md) 的验收方式，明确在“项目初始化与基础骨架”开发完成后，应该如何逐项验证结果是否正确。

本文档的目标不是描述实现方案，而是回答两个问题：

- 做完以后要检查什么
- 看到什么结果才能判断为“通过”

本文档同时参考：

- [第 1 天实施计划](../plan/1.md)
- [代码编写计划](../coding-plan.md)
- [规则文档](../rule.md)

---

## 2. 校验范围

本次校验只覆盖第 1 天骨架建设内容，包括：

- `backend/` 工程是否已初始化成功
- 基础目录结构是否完整
- 配置模块是否可读取、合并并校验
- 日志与错误处理约定是否已接入
- 启动入口与自检脚本是否可以执行

本次明确不校验：

- `Qdrant embedded` 连通性
- `SQLite` 初始化
- `Ollama` 模型调用
- 实际文档索引、检索、问答逻辑
- HTTP API 服务能力

---

## 3. 前置条件

在开始校验前，需满足以下条件：

- 当前仓库已包含 `backend/` 目录
- 本机已安装 `Bun`
- 校验命令在 `backend/` 目录下执行

建议先执行：

```bash
cd backend
bun --version
```

如果 `bun --version` 能正常输出版本号，则可继续后续校验。

---

## 4. 目录结构校验

### 4.1 需要存在的文件与目录

完成第 1 天实现后，至少应存在以下路径：

```text
backend/
├── app.config.json
├── config.ts
├── package.json
├── tsconfig.json
├── data/
├── knowledge/
└── src/
    ├── index.ts
    ├── common/
    │   ├── errors.ts
    │   └── logger.ts
    ├── config/
    │   ├── defaults.ts
    │   ├── load-config.ts
    │   ├── schema.ts
    │   └── types.ts
    ├── ingest/
    ├── llm/
    ├── query/
    ├── scripts/
    │   └── check-config.ts
    ├── server/
    └── store/
```

### 4.2 校验方法

可执行：

```bash
find backend -maxdepth 3 | sort
```

### 4.3 通过标准

- 上述关键文件都存在
- 目录命名与计划文档一致
- 没有把后续天数才需要的复杂实现提前塞入第 1 天骨架

---

## 5. 依赖安装校验

### 5.1 校验命令

```bash
cd backend
bun install
```

### 5.2 通过标准

- 安装过程无报错
- 生成 `bun.lock`
- `package.json` 中仅包含第 1 天需要的最小依赖

### 5.3 失败判定

出现以下情况之一即视为失败：

- 依赖解析失败
- 锁文件未生成
- 引入了与第 1 天目标无关的重型依赖

---

## 6. 类型检查校验

### 6.1 校验命令

```bash
cd backend
bun run typecheck
```

### 6.2 通过标准

- 命令退出码为 `0`
- TypeScript 严格模式下无类型报错

### 6.3 失败判定

出现任意 TS 编译错误即视为失败，必须修复后才能进入后续开发。

---

## 7. 配置加载校验

### 7.1 默认配置校验

执行：

```bash
cd backend
bun run check:config
```

通过标准：

- 命令退出码为 `0`
- 能打印完整配置摘要
- `sources[].path`、`qdrant_path`、`ledger_path` 已转换为绝对路径
- 字段命名与需求文档一致

### 7.2 JSON 覆盖校验

校验方式：

- 修改 `backend/app.config.json` 中至少 1 个字段
- 再次执行 `bun run check:config`

通过标准：

- 输出结果中能体现修改后的值

### 7.3 环境变量覆盖校验

执行：

```bash
cd backend
SERVER_PORT=4001 bun run check:config
```

通过标准：

- 输出中的 `server_port` 为 `4001`
- 说明配置优先级满足：
  `默认值 -> app.config.json -> 环境变量`

---

## 8. 非法配置失败校验

第 1 天除了验证“能启动”，也必须验证“错误配置会失败”。

### 8.1 `chunk_overlap >= chunk_size`

校验目标：

- 当 `chunk_overlap` 大于等于 `chunk_size` 时，配置加载必须失败

通过标准：

- 命令退出非零或明确输出校验失败信息
- 错误内容能指出 `chunk_overlap` 与 `chunk_size` 的约束关系

### 8.2 `similarity_threshold` 超出范围

校验目标：

- 当阈值小于 `0` 或大于 `1` 时，配置加载必须失败

通过标准：

- 错误内容明确指向 `similarity_threshold`

### 8.3 空 `sources`

校验目标：

- 当 `sources` 为空数组时，配置加载必须失败

通过标准：

- 错误内容明确指出至少需要一个来源目录
- 不允许静默回退到默认值

---

## 9. 启动与日志校验

### 9.1 校验命令

```bash
cd backend
bun run dev
```

### 9.2 通过标准

- 启动命令无报错
- 日志中能看到启动成功信息
- 日志至少包含时间戳、级别、模块名

可接受的日志示例：

```text
[2026-04-23T10:00:00.000Z] INFO [bootstrap] skeleton started
```

### 9.3 错误处理通过标准

- 已知错误按统一结构输出
- 未知错误不会被静默吞掉
- 失败时进程退出码非 `0`

---

## 10. 完成定义

当以下项目全部满足时，可判定第 1 天校验通过：

1. `backend/` 已形成最小可运行工程。
2. 类型检查、配置检查、启动命令均可通过。
3. 配置的正常路径与失败路径都已验证。
4. 文档结构满足“有 plan 就有对应 test”的规则。

如果第 1 天实现完成但无法通过本文件中的校验项，则不应视为真正完成。
