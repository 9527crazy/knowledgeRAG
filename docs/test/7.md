## 2. 校验范围

本次校验覆盖：

- Prompt 拼装：`buildChatMessages(question, sources)` 输出 `system + user` 两条消息，且 user 消息包含问题与按编号列出的参考资料
- Ollama Chat 流式：`streamChat(messages)` 能稳定逐 token 吐出文本，无截断 JSON、无悬挂连接
- Chat 服务编排：`streamAnswer(question)` 按 `sources → delta* → done` 顺序产出事件
- 资料不足兜底：`empty: true` 路径下不调用 LLM，直接产出兜底文本
- 演示脚本：`bun run chat:demo` 在终端逐 token 输出，最后打印 `sources`

本次不校验：

- HTTP API / SSE（归属第 8 天）
- 多轮对话上下文（§6.2 第一阶段单轮，明确不实现）
- 文件监听与自动增量更新（归属第 9 天）

## 6. Prompt 拼装校验

### 6.1 校验目标

- 输入非空 `question` 与至少一条 `Source`，输出包含两条消息：`role=system` 与 `role=user`
- system 消息明确"仅根据参考资料回答 / 资料不足时如实回复"等约束
- user 消息以编号列表形式列出 `section_title` / `source_path` / `score` / `chunk_text`
- 输入 `sources.length === 0` 时，`buildChatMessages` 抛 `ValidationError`（防御性边界，正常路径由 `chat-service` 短路）

### 6.2 建议操作

可选择以下任一方式触发：

- 直接在 REPL/脚本中调用 `buildChatMessages("test question", [...])` 并打印
- 设置 `PROMPT_DEBUG=1` 跑 `chat:demo`，让 `chat-service` 在调用 LLM 前把 `messages` 打印到 stderr

### 6.3 通过标准

- 输出消息条数 = 2
- system 消息字符串包含"参考资料"或"仅根据"等关键词
- user 消息包含 `[1]`/`[2]` 编号、`section_title:`、`source_path:`、`score:` 等字段
- 空来源场景能明确失败（不返回空 messages）

---

## 7. Ollama Chat 流式输出校验

### 7.1 校验目标

- `streamChat(messages)` 能从 Ollama 真实拉取 token 并逐段产出
- 网络层面不出现"半截 JSON"导致解析失败
- 当 `done: true` 出现时，AsyncIterable 自然结束

### 7.2 建议操作

执行：

```bash
cd backend
bun run chat:demo "RAG 知识库系统概览的核心目标是什么？"
```

观察 stdout：

- 文本应分多次 chunk 出现（不是一次性整段打印）
- 末尾有完整句号或自然结束
- 不出现裸 `{` 或未解析的 JSON 碎片

### 7.3 通过标准

- 命令退出码为 `0`
- 输出过程中能看到 token / 短句逐步出现
- 终端没有出现 NDJSON 解析错误

---

## 8. Chat 服务事件流校验

### 8.1 校验目标

`chat-service.streamAnswer(question)` 严格按以下顺序吐事件：

1. 一个 `sources` 事件（`items` 与 `runRetrieval().sources` 对齐）
2. 若干个 `delta` 事件（流式 token）
3. 一个 `done` 事件
4. 异常路径：`error` → `done`

### 8.2 建议操作

`chat:demo` 应把不同类型事件分别打印，例如：

```text
[sources] 6 items
[delta] RAG ...
[delta] 系统的核心目标 ...
[done]
```

或采用 JSON-Lines 形式，每行一个事件对象。

### 8.3 通过标准

- `sources` 事件**先于**任何 `delta` 出现
- `done` 事件位于最后
- 没有出现 `delta` 在 `done` 之后

---

## 9. 资料不足兜底校验（不调用 LLM）

### 9.1 校验目标

- 调高 `SIMILARITY_THRESHOLD` 后再次提同一问题，输出兜底文本（如"资料中未找到与该问题直接相关的内容..."）
- 整个过程**不发起 Ollama Chat 调用**

### 9.2 建议操作

```bash
cd backend
SIMILARITY_THRESHOLD=0.99 bun run chat:demo "RAG 知识库系统概览的核心目标是什么？"
```

观察：

- `sources` 事件 `items.length === 0`
- 单个 `delta` 输出兜底文本
- `done` 事件正常结束

可在 Ollama 进程的访问日志（或在 `chat-client` 内部加 `log.info("ollama chat invoked")`）确认空召回路径未触发该日志。

### 9.3 通过标准

- 兜底文本出现且语义清晰
- `chat-client.streamChat` 未被调用（可通过日志或断点确认）
- 命令退出码为 `0`

## 10. 端到端链路校验

### 10.1 校验目标

完成需求 §6.1 描述的完整查询流：

```text
question → embed → retrieve → filter → prompt → LLM → answer + sources
```

### 10.2 建议操作

```bash
cd backend
bun run chat:demo "RAG 知识库系统概览的核心目标是什么？"
```

观察：

- 终端先出现来源列表（含 `section_title` 与 `source_path`）
- 然后流式产出回答
- 回答内容明显基于参考资料，未编造与知识库无关的内容
- 命令在合理时间内完成（首 token ≤ 3 s 为目标，§10 非功能需求；本地机器性能允许少量浮动）

### 10.3 通过标准

- 来源 + 回答 + 结束三段都正常
- 回答语义可读、与问题相关
- 未出现 `error` 事件

## 11. 完成定义

当以下全部满足时，可判定第 7 天校验通过：

1. `prompt-builder.ts`、`chat-client.ts`、`chat-service.ts`、`chat-demo.ts` 已落地。
2. 默认阈值下 `bun run chat:demo` 能产生「来源 + 流式回答 + 结束」三段。
3. 调高阈值至 `0.99` 时，能稳定走兜底路径并不调用 LLM。
4. `bun run typecheck` 通过。
5. 与 [规则文档](../rule.md) 一致：存在 [第 7 天实施计划](../plan/7.md) 与本文件的成对关系。

若第 7 天代码已写完，但仍无法通过本文件中的校验项，则不能视为真正完成，也不宜进入第 8 天 API 与 SSE 联调开发。
