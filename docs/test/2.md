# 第 2 天校验文档：数据层与模型客户端封装

## 1. 文档目的

本文档用于对齐 [第 2 天实施计划](../plan/2.md) 的验收方式，明确在“数据层与模型客户端封装”完成后，应该如何检查依赖封装、存储初始化和健康检查脚本是否真正可用。

本文档主要回答：

- 第 2 天做完后要检查哪些内容
- 哪些现象代表实现通过
- 哪些现象说明仍然不能进入第 3 天开发

本文档同时参考：

- [第 2 天实施计划](../plan/2.md)
- [代码编写计划](../coding-plan.md)
- [规则文档](../rule.md)

---

## 2. 校验范围

本次校验覆盖以下内容：

- `Ollama` 客户端基础能力
- `Qdrant embedded` 初始化能力
- `SQLite` 台账初始化能力
- `document_index` 表结构
- `health-check` 脚本输出

本次明确不校验：

- 文档扫描
- 文档解析
- 切块逻辑
- 向量写入
- 检索与问答主链路

---

## 3. 前置条件

开始校验前，需满足以下条件：

- `backend/` 已存在并可执行 Bun 脚本
- 本机已安装 `Bun`
- 本机已安装并启动 `Ollama`
- 本机已启动 `Qdrant`
- 已拉取模型：
  - `qwen2.5:7b`
  - `bge-m3`

建议先执行：

```bash
ollama list
curl http://127.0.0.1:6333/collections
cd backend
bun --version
```

如果 `ollama list` 无法执行、列表中缺少目标模型，或 `curl http://127.0.0.1:6333/collections` 无法返回结果，则第 2 天健康检查不应判定为通过。

---

## 4. 文件产出校验

### 4.1 需要存在的文件

完成第 2 天实现后，至少应存在以下文件：

```text
backend/src/llm/ollama-client.ts
backend/src/store/qdrant.ts
backend/src/store/ledger.ts
backend/src/scripts/health-check.ts
```

### 4.2 校验方法

可执行：

```bash
find backend/src -maxdepth 3 | sort
```

### 4.3 通过标准

- 上述文件存在
- 目录位置与计划文档一致
- 未把第 3 天及之后的业务逻辑提前塞进这些模块

---

## 5. 依赖安装校验

### 5.1 校验命令

```bash
cd backend
bun install
```

### 5.2 通过标准

- 安装过程无报错
- `bun.lock` 已更新
- 第 2 天新增依赖与 Ollama、Qdrant、SQLite 直接相关

### 5.3 失败判定

出现以下情况之一即视为失败：

- 依赖安装失败
- 引入与第 2 天范围无关的大量额外依赖
- 原生模块依赖安装后不可用

---

## 6. 类型检查校验

### 6.1 校验命令

```bash
cd backend
bun run typecheck
```

### 6.2 通过标准

- 命令退出码为 `0`
- 新增客户端与存储模块无类型错误

---

## 7. Ollama 校验

### 7.1 服务可达性校验

校验目标：

- `health-check` 能识别 Ollama 服务是否可连接

通过标准：

- 输出中有 `ollama.reachable: true`

失败判定：

- 服务不可达却没有报错
- 错误信息无法区分“服务不可达”和“模型不存在”

### 7.2 模型存在性校验

校验目标：

- 能分别检查 `llm_model` 与 `embedding_model`

通过标准：

- 输出中能看到两个模型的就绪状态
- 如果模型不存在，输出应明确为未就绪，而不是笼统失败

---

## 8. Qdrant 校验

### 8.1 初始化校验

执行：

```bash
cd backend
bun run health-check
```

通过标准：

- 输出中可看到 `qdrant.ok: true`
- `qdrant_path` 对应目录已创建或被成功复用
- collection 已被初始化或检测为已存在

### 8.2 失败判定

出现以下情况之一即视为失败：

- 本地目录无法创建
- collection 初始化失败
- 错误信息无法定位到 Qdrant 模块

---

## 9. SQLite 台账校验

### 9.1 初始化校验

执行 `bun run health-check` 后，应能确认：

- `ledger.ok: true`
- `ledger_path` 对应数据库文件已创建
- `document_index` 表存在

### 9.2 表结构校验

建议至少人工确认以下字段已经存在：

- `source_path`
- `doc_id`
- `content_hash`
- `chunk_ids`
- `status`
- `error_msg`
- `indexed_at`
- `updated_at`

### 9.3 通过标准

- 表可通过初始化脚本创建
- 重复执行初始化不会报错
- 结构足以支撑第 4 天增量索引状态机

---

## 10. 健康检查脚本校验

### 10.1 校验命令

```bash
cd backend
bun run health-check
```

### 10.2 通过标准

- 命令可执行
- 输出中同时包含 `ollama`、`qdrant`、`ledger` 三类状态
- 输出格式清晰，适合人工查看
- 任一依赖失败时，脚本会给出明确失败项

### 10.3 推荐结果特征

可接受的输出应至少具备类似结构：

```json
{
  "ollama": {
    "reachable": true,
    "llm_model_ready": true,
    "embedding_model_ready": true
  },
  "qdrant": {
    "ok": true
  },
  "ledger": {
    "ok": true
  }
}
```

---

## 11. 完成定义

当以下项目全部满足时，可判定第 2 天校验通过：

1. 第 2 天新增模块文件已经落位。
2. 依赖安装、类型检查、健康检查脚本均可通过。
3. Ollama、Qdrant、SQLite 三类依赖状态可被清晰识别。
4. `document_index` 表已经具备后续状态机所需的最小结构。
5. 文档结构继续满足“有 plan 就有对应 test”的规则。

如果第 2 天代码已写完，但仍无法通过本文件中的校验项，则不能视为真正完成。
