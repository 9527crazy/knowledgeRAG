# 第 4 天校验文档：切块与增量索引状态机

## 1. 文档目的

本文档用于对齐 [第 4 天实施计划](../plan/4.md) 的验收方式，明确在「切块、文件 hash、增量状态机、SQLite 台账读写」完成后，应如何验证行为是否满足第 5 天 Embedding 与向量入库的前提。

本文档主要回答：

- 第 4 天完成后要检查哪些内容
- 哪些现象代表切块与增量决策已达标
- 哪些问题说明还不能进入第 5 天主链路开发

本文档同时参考：

- [第 4 天实施计划](../plan/4.md)
- [代码编写计划](../coding-plan.md)
- [需求文档](../requirement.md)（§5.2、§5.4）
- [规则文档](../rule.md)

---

## 2. 校验范围

本次校验覆盖以下内容：

- 递归字符切块与配置项：`chunk_size`、`chunk_overlap`、`min_chunk_length`
- 切块产物结构（含 `chunk_id`、`chunk_text`、文档级元数据、`chunk_index` 等计划约定字段）
- 原始文件 **MD5**（十六进制），与台账 `content_hash` 比对逻辑
- 增量索引状态判断：**新增**、**跳过**、**更新**、**删除**
- SQLite `document_index` 的读取与写入（至少能支撑上述判定）

本次明确不校验：

- Ollama Embedding 调用是否成功
- Qdrant 向量写入或删除是否执行（若第 4 天实现为占位，本阶段仅校验台账与意图一致）
- HTTP API、文件监听、查询与问答

---

## 3. 前置条件

开始校验前，需满足以下条件：

- `backend/` 已存在，`bun run typecheck` 可作为常规检查入口
- 第 3 天 ingest 解析与扫描能力可用（`scanSourceFiles`、`parseScannedFile`）
- 第 2 天台账库可初始化，`ledger_path` 配置有效（见 `app.config.json` 或项目默认配置）
- 知识源目录（如 `backend/knowledge/`）中可准备或已存在用于测试的 `.md` / `.txt`

建议额外准备：

- 一段**明显超过** `chunk_size` 的长文本（可放在单个 Markdown 或 TXT 中），用于观察多 chunk 与 overlap
- 一份**台账中已存在**的 `source_path` 记录（可在本阶段手动插入或由一次「新增」验收流程产生），用于做跳过/更新对照

## 6. 文件 Hash 校验

### 6.1 校验目标

- 对磁盘文件原始字节计算 **MD5**，输出稳定十六进制字符串
- 同一路径、同一文件内容多次计算结果一致

### 6.2 建议检查点

- 修改文件任意一个字节后，hash 发生变化
- hash 用于与台账 `content_hash` 比较；比较逻辑大小写约定一致（建议统一为小写 hex）

### 6.3 通过标准

- 「未改文件 → hash 不变」「改内容 → hash 变」可复现
- 状态机「跳过」分支依赖 hash 一致时可稳定触发（见第 8 节）

---

## 7. 切块逻辑校验

### 7.1 校验目标

- 长文本可被切成多个 chunk，数量与配置大致可解释（窗口上限、分隔符优先级）
- `chunk_overlap` 在相邻块之间生效；`chunk_size` 为上界约束
- 长度低于 `min_chunk_length` 的片段被丢弃（边界是否包含等于阈值以代码注释为准，验收时人工对照一次）

### 7.2 建议检查点

- 每个 chunk 具备唯一 `chunk_id`（通常为 UUID）
- `chunk_index` 在单文档内从 0 递增（或项目约定另一种单调规则，须在代码或文档中单点说明）
- `doc_id`、`source_path`、`section_title` 等元数据不因切块而丢失可追溯性

### 7.3 建议操作

若有演示脚本（如计划中可选的 `index-plan-demo` 或等价命令），执行后检查输出的 chunk 数量与样例长度是否匹配预期。

### 7.4 通过标准

- 调整 `chunk_size` / `chunk_overlap` / `min_chunk_length` 后，切块数量或边界随之合理变化
- 同一 `ParsedDocument` 与同一配置下，重复运行切块结果**稳定**（确定性）

---

## 8. 增量状态机与台账校验

### 8.1 校验目标

- **新增**：扫描结果中存在、`document_index` 中无对应 `source_path` → 判定为需索引（或等价动作名）
- **跳过**：台账存在且 `content_hash` 与当前文件 MD5 一致 → 不重复切块与无意义更新（或仅允许约定内的轻量字段）
- **更新**：台账存在且 hash 不一致 → 判定为更新，并能读取旧 `chunk_ids`（供第 5 天删旧向量）
- **删除**：台账中存在某 `source_path`，但当前扫描集合中已无该文件 → 判定删除，并能取出待删除的 `chunk_ids`

### 8.2 建议检查点

- 扫描集合与台账 diff 使用与第 3 天一致的绝对路径口径，避免相对/绝对混用导致误判
- `status`、`chunk_ids`、`content_hash`、`updated_at`（若使用）在写入后可通过查询验证

### 8.3 通过标准

- 上述四类场景至少各完成一次人工或脚本辅助的对照实验，现象与 [需求文档](../requirement.md) §5.2 描述一致
- 删除场景下，台账记录被删除或按约定标记为已删除，且与「孤儿路径」判定一致

---

## 9. 与第 5 天的边界校验

### 9.1 校验目标

确认第 4 天交付物为第 5 天提供稳定输入，而非提前实现半套入库：

- 切块 UUID 列表可写入台账 `chunk_ids`（JSON 数组字符串等形式）
- 更新/删除路径能拿到旧 UUID 列表，无需在第 5 天重新猜测

### 9.2 通过标准

- 台账中存的 `chunk_ids` 与切块模块输出的 `chunk_id` 列表可对应（顺序或索引约定明确）
- 明知尚未写入 Qdrant 时，`status` 语义在代码注释或 README 中与第 5 天衔接说明一致，验收不产生歧义

---

## 10. 完成定义

当以下项目全部满足时，可判定第 4 天校验通过：

1. `chunker.ts`、`hash.ts`、`index-state-machine.ts` 已落地，台账读写可满足状态机需求。
2. 切块参数与过滤规则可通过配置调节并通过第 7 节校验。
3. MD5 与四类增量意图判定可通过第 6、8 节校验。
4. `bun run typecheck` 通过。
5. 与 [规则文档](../rule.md) 一致：存在 [第 4 天实施计划](../plan/4.md) 与本校验文档的成对关系。

若第 4 天代码已写完，但仍无法通过本文件中的校验项，则不能视为真正完成，也不宜进入第 5 天向量入库主链路。
