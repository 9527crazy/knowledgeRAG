## 2. 校验范围

本次校验覆盖：

- HTTP API：`POST /api/chat` 可接收 `question` 请求
- SSE：响应头正确、可以持续收到事件帧
- 事件定义：`delta` / `sources` / `done` / `error` 事件可复现，且数据结构符合需求 §8.1
- 请求校验：缺参/空字符串等返回明确错误
- 端到端联调：服务启动后通过 curl 完成一次提问并收到完整事件序列

本次不校验：

- `GET /api/status`、`POST /api/reindex`（第一阶段可做但非第 8 天必需）
- 多轮上下文、鉴权、限流
- 文件监听与自动增量更新（归属第 9 天）

## 7. `POST /api/chat` SSE 协议校验

### 7.1 校验目标

- `Content-Type` 为 `text/event-stream`
- 连接不被一次性返回/缓冲（能看到分段输出）
- 至少能看到 `delta` 与 `done` 事件
- 有召回时能看到 `sources` 事件（items 非空）

### 7.2 建议操作（curl）

有召回问题（示例）：

```bash
curl -N \
  -H "Content-Type: application/json" \
  -d '{"question":"RAG 知识库系统概览的核心目标是什么？"}' \
  http://localhost:3000/api/chat
```

期望输出（顺序以实现为准，至少包含这些事件）：

```text
event: sources
data: {"items":[...]}

event: delta
data: {"text":"..."}

event: done
```

### 7.3 通过标准

- `sources`（如有）/ `delta` / `done` 事件均出现
- `data` 是单行 JSON 且能被解析
- `done` 事件在末尾出现且只出现一次

## 8. 请求校验与错误事件校验

### 8.1 校验目标

- 缺 `question` / 非 string / 空字符串：返回 400（非 SSE）或返回 `error` 事件（若选择 SSE 统一错误）二选一，但必须**稳定可解释**
- 运行时异常：必须出现 `error` 事件，并最终 `done`

### 8.2 建议操作

缺字段：

```bash
curl -i \
  -H "Content-Type: application/json" \
  -d '{}' \
  http://localhost:3000/api/chat
```

空字符串：

```bash
curl -i \
  -H "Content-Type: application/json" \
  -d '{"question":"   "}' \
  http://localhost:3000/api/chat
```

### 8.3 通过标准

- 错误语义明确：能知道是“请求参数错误”还是“服务内部错误”
- 不会出现悬挂连接（无响应或永不结束）

---

## 9. 空召回短路（不调用 LLM）联调校验

### 9.1 校验目标

调高 `SIMILARITY_THRESHOLD`（例如 `0.99`）后：

- SSE 仍能返回 `sources(items=[])` + `delta(兜底文本)` + `done`
- 服务器日志中不应出现 `chat-client` 的 “ollama chat invoked”（若你按第 7 天实现保留该日志）

### 9.2 建议操作

启动服务前设置环境变量（示例）：

```bash
cd backend
SIMILARITY_THRESHOLD=0.99 bun run dev
```

然后请求：

```bash
curl -N \
  -H "Content-Type: application/json" \
  -d '{"question":"RAG 知识库系统概览的核心目标是什么？"}' \
  http://localhost:3000/api/chat
```

### 9.3 通过标准

- `sources` 事件 items 为空数组
- 输出兜底文本
- `done` 正常出现

## 10. 完成定义

当以下全部满足时，可判定第 8 天校验通过：

1. `app.ts`、`sse.ts`、`routes/chat.ts` 落地，并能对外提供 `POST /api/chat` SSE。
2. curl 可收到 `delta`/`sources`/`done`/`error` 事件（至少覆盖正常与异常两类）。
3. 请求校验清晰，空召回短路可复现且不调用 LLM。
4. `bun run typecheck` 通过。
5. 与 [规则文档](../rule.md) 一致：存在 [第 8 天实施计划](../plan/8.md) 与本文件的成对关系。

若第 8 天代码已写完，但仍无法通过本文件中的校验项，则不能视为真正完成，也不宜进入第 9 天文件监听与增量更新开发。
