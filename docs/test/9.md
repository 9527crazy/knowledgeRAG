# 第 9 天校验文档：文件监听、增量更新与异常补强

本文档用于对齐 [第 9 天实施计划](../plan/9.md) 的验收方式，明确在实现「目录监听 + 增量更新 + 启动补扫 + 异常补强」后应如何检查。验收通过意味着系统具备基础的“资料变更自动生效”能力，可进入第 10 天测试与文档收口。

## 2. 校验范围

本次校验覆盖：

- watcher：对 `config.sources` 目录监听生效
- 增量索引：新增/修改/删除文件能驱动索引更新
- 启动补扫：服务重启能补扫离线期间遗漏变更
- 异常可观测：失败能在台账与日志中定位

本次不校验：

- `/api/status` 与 `/api/reindex`（第 10 天收口）
- 并发索引调度（第一阶段可先串行）
- 大规模性能压测

## 6. watcher 端到端校验（新增/修改/删除）

### 6.1 准备一个可观察的测试文件

在 `backend/knowledge/` 下准备一个测试文件（示例路径）：

```text
backend/knowledge/watch-test.md
```

内容建议包含一个清晰关键词（如 `WATCH_KEYWORD_1`），便于后续通过检索验证更新是否生效。

### 6.2 启动 watcher（以实现为准）

示例（仅示意，按你的脚本/入口为准）：

```bash
cd backend
bun run dev
```

或（如果你实现了独立脚本）：

```bash
cd backend
bun run watch:index
```

### 6.3 新增文件应触发索引

操作：

- 创建 `watch-test.md`
- 观察日志中出现 add/indexing/done 相关信息

通过标准：

- SQLite 台账中出现该文件记录，状态最终为 `done`
- Qdrant 中能检索到包含 `WATCH_KEYWORD_1` 的 chunk（可通过 `query:demo` 或后续 API/脚本验证）

### 6.4 修改文件应触发更新

操作：

- 修改 `watch-test.md`，将关键词改为 `WATCH_KEYWORD_2`
- 观察日志中出现 update 相关信息

通过标准：

- 新关键词可被检索到
- 旧关键词检索结果显著减少或消失（取决于你的检索阈值与文本内容）

### 6.5 删除文件应触发删除

操作：

- 删除 `watch-test.md`
- 观察日志中出现 remove 相关信息

通过标准：

- 台账中该文件记录被删除或不再可查
- 向量库中对应 chunk 不再可召回（可通过检索验证）

## 7. 启动补扫校验（重启后补齐）

目标：模拟服务离线期间发生变更，重启后能够补扫。

步骤：

1. 停止服务/Watcher
2. 离线期间创建或修改一个知识文件（例如新增 `backend/knowledge/offline-add.md`）
3. 重启服务/Watcher

通过标准：

- 重启后无需人工触发，即可完成对新增/修改文件的索引
- 日志中能看到一次“启动扫描/补扫”相关记录（以实现日志为准）

---

## 8. 异常可观测性校验（失败可定位）

目标：触发一次可控失败，确认 `failed` 能落库且日志可解释。

可选方案（任选一种，按环境可操作性选择）：

- 临时把 `OLLAMA_BASE_URL` 指向不可用地址后启动 watcher 并触发索引
- 或用极端内容触发解析失败（不建议，优先用 Ollama 不可达）

通过标准：

- 台账中对应文件状态为 `failed`
- `error_msg` 有明确原因（例如网络不可达/请求失败）
- 恢复依赖后重新触发索引能回到 `done`

---

## 9. 完成定义

当以下全部满足时，可判定第 9 天校验通过：

1. watcher 可稳定处理新增/修改/删除三类事件并完成增量更新。
2. 重启补扫可复现并能补齐离线期间变更。
3. 失败可在台账与日志中定位，且恢复依赖后可重新索引成功。
4. `bun run typecheck` 通过。
