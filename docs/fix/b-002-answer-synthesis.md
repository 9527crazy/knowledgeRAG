# knowledgeRAG 回答拟合增强方案

## 1. 根因结论

当前模型回答看起来像“直接返回检索文案”，主要原因不是 RAG 链路没有接通，而是回答生成阶段的约束组合让模型过于保守。

现有链路已经完成：

```text
用户问题
-> Query Planner 生成检索 query
-> Embedding
-> Qdrant 检索
-> sources 映射
-> buildChatMessages(question, sources)
-> Ollama Chat
-> SSE delta 输出
```

问题集中在两个点：

1. `backend/src/query/prompt-builder.ts` 的事实约束较强：
   - `system` 要求“仅根据参考资料回答”“不得编造”“尽量引用来源”；
   - 这些约束能抑制幻觉，但也会让模型倾向于贴近原文；
   - 同时 Prompt 没有明确要求模型对多段检索结果做归纳、去重、重组和改写；
   - `chunk_text` 会以原文形式完整进入 Prompt，模型在强事实约束下更容易直接摘抄。
2. 当前 `llm_temperature = 0.1`：
   - 事实稳定性较好；
   - 但表达重组主动性偏低；
   - 与强事实约束叠加后，模型更容易采用最保守的“原文复述/摘录”策略。

因此模型并不是没有使用检索内容，而是在“强依据约束 + 低温度 + 缺少拟合指令”的组合下，采用了最保守的“照资料复述/摘录”策略。

---

## 2. 最小修复：补充回答拟合 Prompt

### 2.1 目标

让模型在不脱离参考资料的前提下，把检索片段拟合成面向用户问题的答案：

- 先理解用户问题；
- 从多个检索片段中提取相关事实；
- 合并重复信息；
- 用自己的话组织答案；
- 不整段照抄原文；
- 资料不足时明确说明不足；
- 保留来源引用，便于追溯。

### 2.2 建议 Prompt

将 `backend/src/query/prompt-builder.ts` 中的 `system.content` 调整为更明确的回答策略。注意不是放松“仅依据资料”的约束，而是在该约束下补充“如何归纳生成”的指令：

```text
你是一个本地知识库问答助手。
你必须仅根据我提供的【参考资料】回答，不得编造或凭空推测。

回答策略：
1. 先判断用户问题真正想问什么，再从参考资料中提取相关事实。
2. 对多个参考片段进行归纳、合并、去重和重组，用自然语言回答用户问题。
3. 不要整段照抄参考资料；除非用户要求原文，否则只保留必要短引用。
4. 如果参考资料之间有重复或碎片化内容，请整理成清晰条目。
5. 如果参考资料不足以回答问题，请明确说明资料不足，并指出缺少什么信息。
6. 回答末尾列出使用到的来源，包含 section_title 与 source_path。
```

`user.content` 仍保持：

```text
# 用户问题
...

# 参考资料
...
```

## 3. 输出格式建议

为减少“原文贴回”的概率，可以要求固定结构：

```text
结论：
...

依据：
- ...
- ...

来源：
- section_title: ... | source_path: ...
```

注意：固定格式不一定要长期保留。MVP 阶段建议先加上，便于观察模型是否完成了“归纳生成”。

---

## 4. 可选增强

### 4.1 对参考资料增加边界标识

当前每段资料格式为：

```text
[1] section_title: ...
    source_path: ...
    score: ...
    --
    chunk_text
```

可以改成更清晰的边界：

```text
<source id="1">
section_title: ...
source_path: ...
score: ...
content:
...
</source>
```

这样模型更容易区分“资料内容”和“需要生成的答案”。

### 4.2 控制每段 chunk 长度

如果 `chunk_text` 过长或包含大量无关内容，模型更容易摘抄。可以在进入 Prompt 前做轻量裁剪：

- 单个 source 最多保留固定字符数；
- 优先保留命中片段所在段落；
- 后续再考虑检索后摘要压缩。

### 4.3 适度调整温度

当前 `llm_temperature = 0.1`。如果回答仍过于机械，可尝试：

```json
"llm_temperature": 0.2
```

不建议一开始调太高，否则事实问答稳定性会下降。

## 5. 改动模块

| 模块 | 改动 |
| --- | --- |
| `backend/src/query/prompt-builder.ts` | 在事实约束基础上补充拟合策略，明确要求归纳、合并、改写、去重 |
| `backend/src/query/prompt-builder.ts` | 可选：调整 `formatSourceBlock`，增加 source 边界标识 |
| `backend/app.config.json` | 可选：将 `llm_temperature` 从 `0.1` 调整到 `0.2` |
| `backend/tests` | 建议补充 Prompt 构造单测，避免后续误删关键指令 |

---

## 6. 实施顺序

**阶段一（立即）：**

- 只改 `prompt-builder.ts` 的系统提示词，补充拟合策略。
- 保持检索、排序、前端展示逻辑不变。
- 打开 `PROMPT_DEBUG=1`，确认实际发给 Ollama 的 messages 符合预期。

**阶段二（短期）：**

- 调整 `formatSourceBlock`，给每个 source 增加显式边界。
- 增加 Prompt builder 单测，断言包含“归纳、合并、去重、不要整段照抄、来源”等关键约束。

**阶段三（按效果决定）：**

- 试调 `llm_temperature = 0.2`，缓解低温度导致的过度保守。
- 针对长 chunk 做进入 Prompt 前的裁剪或摘要压缩。

---

## 7. 验收口径

启动后端时打开 Prompt 调试：

```bash
PROMPT_DEBUG=1 bun run dev
```

调用问答接口：

```bash
curl -N -H "Content-Type: application/json" \
  -d '{"question":"根据检索到的资料，总结这个项目当前的 RAG 问答链路是什么？"}' \
  http://localhost:3000/api/chat
```

期望：

- 后端日志中能看到参考资料进入 Prompt；
- 回答不是整段复制 `chunk_text`；
- 回答能围绕用户问题进行归纳和组织；
- 多个来源内容重复时，答案会合并去重；
- 回答末尾能列出使用到的 `section_title` 与 `source_path`；
- 资料不足时不会硬答，而是说明缺少哪些信息。

---

## 8. 风险与边界

- Prompt 拟合策略只能改善“有正确资料时的生成方式”，不能解决“检索结果本身不相关”的问题。
- 如果召回片段质量差，模型仍然只能基于差资料回答。
- 如果用户要求“原文是什么”，此时允许摘录原文；否则默认应总结改写。
