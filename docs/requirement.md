一、目标定义（必须明确）

目标：
构建一个本地知识系统，实现：

用自然语言查询本地资料
自动检索相关内容
基于上下文生成准确回答

技术约束：

全部本地运行（无外部 API）
使用 Qwen 作为大模型
轻量、可维护、无复杂依赖
二、核心能力（功能需求）
1）数据接入（Ingest）

目标：把本地文件变成可检索数据

支持：

txt / md（第一阶段）
代码文件（可选）

能力要求：

扫描目录 /data
自动读取文件内容
支持手动触发更新
2）文档处理（Processing）

目标：让数据适合检索

功能：

文本切分（chunk）
去除无效内容（空行、重复）

规则：

chunk size：300~600
overlap：50~100
3）向量化（Embedding）

目标：让文本可进行语义搜索

能力：

文本 → 向量
查询 → 向量

要求：

本地模型
无网络依赖
4）向量存储（Storage）

使用：

FAISS

能力：

保存向量数据
加载向量数据
支持重建索引

数据结构要求：

{
  "text": "chunk内容",
  "source": "文件路径",
  "id": "唯一标识"
}
5）检索（Retrieval）

目标：找到最相关内容

能力：

输入问题
返回 Top-K 相关片段

参数：

K = 3~5
6）生成（Generation）

使用：

Ollama 调用 Qwen

能力：

拼接 context
构建 prompt
输出答案

约束：

不允许编造
必须基于上下文
7）查询接口（Query）

形式：

CLI（第一阶段）
或简单 HTTP API

输入：

{
  "question": "xxx"
}

输出：

{
  "answer": "xxx",
  "sources": [...]
}
三、非功能需求
1）性能
单次查询 < 3 秒
支持 < 10 万 chunk
2）资源
内存：16GB+
CPU优先（可无GPU）
3）稳定性
向量库持久化
崩溃可恢复
4）可维护性
模块清晰（ingest / query 分离）
支持重新构建索引
四、系统模块划分
1. ingest模块
   - 文件读取
   - chunk切分
   - embedding
   - 写入FAISS

2. retriever模块
   - 向量检索

3. llm模块
   - 调用Qwen

4. service模块
   - 串联整个流程
五、目录结构（标准）
/rag
 ├── data/        # 原始数据
 ├── db/          # 向量库
 ├── ingest.js
 ├── query.js
 ├── retriever.js
 ├── llm.js
 └── config.js
六、关键流程（必须实现）
流程1：数据入库
扫描文件
 → 切分
 → embedding
 → 写入FAISS
流程2：查询
输入问题
 → embedding
 → 检索Top-K
 → 拼接context
 → 调用Qwen
 → 输出答案
七、边界（避免过度设计）

当前版本不做：

❌ 多用户系统
❌ 权限管理
❌ 在线服务
❌ Agent 自动执行
❌ Skill 抽象
❌ 多模型调度
八、验收标准（你可以用这个判断是否完成）

满足以下条件即“完成”：

能导入一批 md 文档
能提问并返回合理答案
回答明显优于“纯模型回答”
本地完全离线运行
数据更新后能重新索引
九、后续演进（不是现在做）

当你用一段时间后再考虑：

增量更新（监听文件变化）
简单 UI（Electron）
标签过滤
rerank优化
Skill结构化
十、一句话总结

这个需求的本质是：
把“个人笔记搜索”升级为“可对话的本地知识系统”
