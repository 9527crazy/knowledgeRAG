## 后续需求（产品输入）

> 尚未实现的能力描述，供后续规划与实现引用；与「未修复」中的缺陷条目区分。

### REQ-001 Plan 模式（解题计划先行）

- **目标**：在常规「直接回答」之外，支持一种 **Plan 模式**：模型**先不输出最终答案**，而是先生成一份可读的 **「解题计划」**（步骤、检索意图、需核对的事实点等），再按计划推进或经用户确认后继续。
- **预期体验**：用户可先看计划是否合理，再决定是否执行、改写计划或进入正式作答；降低「一把梭」幻觉与跑题风险。
- **待澄清**：Plan 与最终回答是否分两轮 API、是否在前端分栏展示、是否与现有 RAG 检索管线如何衔接（计划中是否暴露引用意图）。

### REQ-002 聊天记录保存与蒸馏（输入人格分析 + 输出总结）

- **目标**：对会话做 **持久化**，并在后台对内容进行 **蒸馏累积**：对用户输入侧做 **人格 / 风格相关分析**（或用户画像信号），对助手输出侧做 **总结提炼**，形成可随时间更新的摘要与偏好信号。
- **行为约束**：每次用户发送消息后，在 **后台静默** 执行分析与更新，**不阻塞**主对话流；需定义存储位置、隐私边界与累积策略（覆盖式 / 滑动窗口 / 定期合并）。
- **待澄清**：「人格分析」的具体维度（正式度、领域偏好、指令风格等）、蒸馏产物面向何处消费（仅服务端路由 / 可选展示给用户）、与现有用户认证或匿名会话的关系。
